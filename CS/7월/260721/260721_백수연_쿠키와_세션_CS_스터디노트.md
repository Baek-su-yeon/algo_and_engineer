# 쿠키(Cookie)와 세션(Session) — CS 면접 스터디 노트

> 참고 자료: [쿠키(Cookie)와 세션(Session)](https://junhyunny.github.io/information/cookie-and-session/), [쿠키(Cookie) 심층 분석](https://junhyunny.github.io/information/security/cookie-attributes/)

---

## 1. 배경 — 왜 쿠키와 세션이 필요한가

### HTTP의 무상태(Stateless) 특성
- HTTP는 각 요청을 이전 요청과 무관한 독립적인 트랜잭션으로 취급한다.
- **독립적이라는 의미**: 서버가 응답할 때 "누가 요청했는지"를 몰라도, 요청 자체에 담긴 정보(URL, 메서드 등)만으로 응답할 수 있는 상황을 말한다.
  - 예: `GET /about.html` → 누가 요청하든 항상 동일한 내용을 응답. 이전 요청을 기억할 필요가 없음.
- **초기 HTTP(문서 전달 용도)에서는 문제 없었던 이유**: 응답 내용이 요청자가 누구인지와 무관하게 항상 동일해도 됐기 때문.
- **현재 문제가 되는 이유**: 로그인 상태 유지, 장바구니 유지처럼 "누가 요청했는지"에 따라 다른 응답을 줘야 하는 경우가 많아짐. 하지만 HTTP 프로토콜 자체엔 이런 기능이 없음.
- **해결책**: 세션(서버가 사용자별 상태를 저장) + 쿠키(그 사용자를 식별할 세션 ID를 브라우저가 들고 다님)를 조합.

### 쿠키-세션은 왜 "묶여서" 다뤄지는가
- 세션 단독으로는 작동하지 않는다 — 서버에 세션(저장 공간)을 만들어도, 클라이언트가 매 요청마다 "저 아까 그 사람이에요"라고 증명할 방법이 없으면 서버는 그 세션을 찾을 수 없다.
- 쿠키만 있고 세션이 없으면 — 그냥 클라이언트가 들고 있는 데이터일 뿐, "로그인 상태를 서버가 기억한다"는 기능이 성립하지 않는다.
- 즉 **쿠키(전달 수단) + 세션(저장 장소) = 하나의 상태 유지 메커니즘**을 이루는 두 부품.
- HTTPS/Secure/HttpOnly/SameSite 등은 이 메커니즘을 안전하게 지키는 **별개의 보안 계층**이지, 쿠키-세션이 묶이는 근본 이유는 아님.

## 2. 클라이언트 vs 서버 저장 공간

| 위치 | 저장 공간 |
|---|---|
| 클라이언트(브라우저) | 로컬 스토리지, 세션 스토리지, 쿠키 |
| 서버 | 데이터베이스, 캐시, 세션 |

> ⚠️ **주의**: "세션 스토리지"(브라우저, 클라이언트 측)와 "세션"(서버 측)은 이름이 비슷하지만 완전히 다른 개념.

### 브라우저 저장 공간 비교

| 구분 | 로컬 스토리지 | 세션 스토리지 | 쿠키 |
|---|---|---|---|
| 유지 기간 | 명시적으로 삭제 전까지 영구 | 탭/창 닫으면 삭제 | 설정한 만료 시간까지 (없으면 브라우저 종료 시) |
| 저장 범위 | 같은 오리진이면 탭 간 공유 | 탭 단위로 독립 | 도메인/경로(Domain, Path) 기준 공유 |
| 서버 전송 여부 | 전송 안 됨 | 전송 안 됨 | **모든 요청마다 자동 전송** |
| 용량 | 5~10MB | 5~10MB | 개당 약 4KB |
| 보안 옵션 | 없음 (XSS에 취약) | 없음 (XSS에 취약) | HttpOnly, Secure, SameSite 지원 |
| 주 용도 | 사용자 설정, 테마 등 장기 데이터 | 폼 임시 입력값, 탭 단위 상태 | 서버와 주고받아야 하는 값 (세션 ID 등) |

## 3. 쿠키(Cookie)

### 정의
서버가 브라우저에 보내는 작은 데이터 조각(이름-값 쌍). 브라우저가 저장했다가 같은 서버로 요청할 때마다 자동으로 함께 전송한다. 개당 약 4KB까지 저장 가능.

### 생성/전달 흐름
1. 브라우저가 서버에 요청
2. 서버가 응답 헤더에 `Set-Cookie: 이름=값` 형태로 쿠키 전달
3. 브라우저가 `Set-Cookie` 헤더를 보고 쿠키 저장소에 저장 (도메인, 경로, 이름 조합이 키)
4. 이후 같은 도메인 요청마다 브라우저가 자동으로 `Cookie: 이름=값` 헤더에 담아 전송
5. 서버는 쿠키 값을 보고 사용자를 식별해 처리

- 요청은 URL 입력, 링크 클릭, 폼 제출, JS의 fetch/XHR, `<img>`/`<script>` 리소스 로딩 등 다양한 경로로 발생하며, 어떤 방식이든 해당 쿠키가 있으면 자동으로 실림.

### 주요 속성(Attributes)

| 속성 | 설명 |
|---|---|
| `Domain` | 쿠키가 종속된 도메인. 서브도메인도 포함 적용 |
| `Path` | 쿠키가 유효한 URL 경로. **서버가 명시적으로 재설정하지 않는 한 자동 반영 안 됨** (URL 구조가 바뀌면 개발자가 직접 갱신해야 함) |
| `Expires` / `Max-Age` | 만료 시점 (날짜 지정 vs 발급 시점부터 경과 시간 지정) |
| `Secure` | HTTPS 요청에서만 전송 (localhost 제외) |
| `HttpOnly` | JavaScript(`document.cookie`)로 접근 불가 → XSS 방어 |
| `SameSite` | 크로스 사이트 요청에 쿠키를 실어 보낼지 제어 (`Strict`/`Lax`/`None`) → CSRF 방어 |

- **SameSite=None이 필요한 경우**: SSO, 결제/PG사 iframe, 광고·분석 트래킹, 유튜브 등 임베드 서비스처럼 정당하게 크로스 사이트로 쿠키를 보내야 하는 경우. 대신 `Secure` 속성을 필수로 함께 사용해야 함.

### 용도
- 세션 관리 (로그인 상태, 장바구니 등)
- 개인화 (사용자 설정, 테마)
- 추적 (사용자 행동 분석, 광고 타겟팅)

### 쿠키에 민감 정보를 저장하면 안 되는 이유
- 클라이언트(사용자 PC)에 저장되어 접근이 통제되지 않는 공간
- HttpOnly 없으면 XSS로 탈취 가능 (`document.cookie`로 읽힘)
- HTTPS 안 쓰면 네트워크 전송 중 탈취 가능
- → 그래서 쿠키에는 정보 자체가 아니라 **의미 없는 랜덤 값(세션 ID)**만 저장. 탈취돼도 서버가 세션을 즉시 만료시키면 피해를 막을 수 있음.

### HTTPS 없이 쿠키를 쓰면
- HTTP는 평문 통신 → 같은 네트워크의 공격자가 패킷을 가로채(스니핑) 세션 ID를 그대로 읽을 수 있음
- 탈취한 세션 ID를 자기 브라우저에 심으면 로그인 없이 남의 계정 행세 가능 (**세션 하이재킹**)
- `Secure` 속성으로 HTTPS 요청에서만 쿠키가 전송되게 하면 이 위험을 차단

## 4. 세션(Session)

### 정의
서버 쪽 저장 공간. 서버가 각 사용자별로 필요한 정보를 기억하기 위해 사용 (인-메모리 / DB / 캐시 등으로 구현).

### 생성 흐름
1. 클라이언트가 서버에 요청
2. 서버는 요청에 세션 ID가 없거나, 있어도 해당 세션 데이터가 없으면 "새로운 사용자"로 판단
3. 이 사용자를 위한 저장 공간을 세션 영역에 생성 (로그인 정보, 권한 등 저장)
4. 이 공간을 찾을 수 있는 **세션 ID** 발급
5. 응답 시 세션 ID를 `Set-Cookie` 헤더에 담아 클라이언트에 전달

### 서버가 쿠키 값으로 사용자를 식별하는 과정
- 브라우저가 세션 ID를 쿠키로 저장 → 다음 요청부터 자동으로 `Cookie: JSESSIONID=...` 형태로 전송
- 서버는 쿠키 값(세션 ID)을 꺼내 세션 저장소에서 조회
- 있으면 → 해당 사용자 정보로 식별 / 없거나 만료됐으면 → 신규(또는 만료) 사용자로 처리
- **쿠키에는 사용자 정보가 없고, 정보를 찾을 "열쇠"(세션 ID)만 있음.** 실제 데이터는 서버 세션 저장소에 있음.

| | 쿠키 | 세션 |
|---|---|---|
| 위치 | 클라이언트 | 서버 |
| 저장 내용 | 세션 ID(열쇠) | 실제 사용자 데이터 |
| 역할 | 열쇠를 들고 다니는 배달부 | 열쇠로 찾는 금고 |

### 세션을 인메모리가 아닌 Redis 등 외부 캐시에 저장하는 이유
- **다중 서버 문제**: 로드밸런서 뒤에 서버가 여러 대면 요청마다 다른 서버로 갈 수 있음. A 서버에 저장된 세션을 B 서버는 모름 → 로그인 풀린 것처럼 보임.
- **서버 재시작 시 소실**: 인메모리는 프로세스 종료 시 함께 사라짐.
- Redis 같은 외부 저장소를 쓰면 어떤 서버든 같은 저장소를 조회해 세션 공유 가능, 재시작에도 유지됨.
- (대안: Sticky Session — 특정 사용자를 항상 같은 서버로 보냄. 다만 트래픽 쏠림 위험이 있어 중앙 저장소 방식을 더 선호)

### 세션이 많아질 때 메모리 부담
- 사용자 증가 → 세션 데이터 누적 → 애플리케이션 서버 메모리 부담
- 별도 저장소(Redis 등)로 분리해 부담을 옮기고, 만료 시간(TTL)을 짧게 설정해 오래된 세션 자동 정리

## 5. 세션 기반 인증 vs 토큰(JWT) 기반 인증

| | 세션 기반 | JWT(토큰) 기반 |
|---|---|---|
| 사용자 정보 저장 위치 | 서버(세션 저장소) | 클라이언트가 든 토큰 자체 |
| 서버 상태 | 사용자별 상태 기억 (stateful) | 아무것도 기억 안 함 (stateless) |
| 검증 방법 | 세션 ID로 저장소 조회 | 토큰 서명 검증만, 조회 불필요 |
| 서버 확장성 | 여러 서버 간 세션 공유 필요 | 서버가 각자 검증만 하면 됨 → 확장 쉬움 |
| 즉시 무효화 | 서버에서 세션 삭제하면 즉시 끝 | 만료 전까지 유효 → 즉시 무효화 어려움 |

### JWT를 많이 쓰는 이유
- 세션 저장소 조회가 필요 없어 서버 수평 확장에 유리 (마이크로서비스 구조에서 특히)
- 모바일 앱, 다른 도메인 API 서버 등 쿠키를 자연스럽게 못 쓰는 환경에서도 헤더에 토큰만 넣으면 됨

### 세션이 여전히 쓰이는 이유
- 즉시 로그아웃/강제 만료가 쉬움 (JWT는 블랙리스트 등 별도 장치 없이는 즉시 무효화가 어려움) → 은행 등 보안 민감 서비스에서 선호
- 구현이 상대적으로 단순

> 결론: "무조건 JWT가 낫다"가 아니라 **확장성이 중요하면 JWT, 즉각적 제어가 중요하면 세션**인 트레이드오프.

### 로컬 스토리지에 토큰을 저장하면 안 되는 이유
- 로컬 스토리지는 HttpOnly 같은 보호 장치가 없어 JS로 언제든 읽힘 (`localStorage.getItem(...)`)
- XSS 취약점이 있으면 악성 스크립트가 토큰을 그대로 탈취 가능
- 반면 HttpOnly 쿠키는 XSS가 터져도 JS로 값을 못 읽음 → JWT는 로컬 스토리지보다 **HttpOnly 쿠키 저장이 더 안전**하다는 의견이 많음 (단, 이 경우 CSRF 대비를 위해 SameSite도 함께 설정 필요)

## 6. 로그아웃 / 자동 로그인 구현

### 로그아웃
- **세션 기반**: 서버가 해당 세션 ID의 세션 데이터를 저장소에서 삭제. 클라이언트는 쿠키를 만료/삭제 처리.
- **JWT 기반**: 토큰 자체를 서버가 무효화하기 어려워, 보통 클라이언트가 토큰을 삭제. 더 엄격하게 하려면 서버가 "무효화된 토큰 블랙리스트"를 관리하며 매 요청마다 대조.

### 자동 로그인
- **세션 만료 시간 연장 방식**: 세션/쿠키의 만료 시간을 길게 설정(예: 30일). 탈취 시 위험 기간도 길어지는 단점.
- **Remember-me 토큰 방식**: 로그인 세션과 별개로 긴 만료 기간의 별도 토큰을 쿠키에 저장. 원래 세션이 만료돼도 이 토큰이 유효하면 서버가 새 세션을 자동 발급. 보안을 위해 보통 **한 번 사용 시 새 값으로 교체(토큰 로테이션)**해 탈취 시 재사용을 방지.

## 7. 쿠키 동의 팝업은 왜 뜨는가 (트래킹 목적)

### 배경 — 법적 이유
- EU GDPR/ePrivacy 지침 등: **사용자를 추적하거나 개인정보를 수집하는 목적의 쿠키는 명시적 동의 없이 심으면 안 됨**. 그래서 사이트가 "쿠키 저장해도 될까요?"라고 먼저 물어봐야 함.

### 쿠키 종류별 동의 필요 여부

| 종류 | 예시 | 동의 필요 여부 |
|---|---|---|
| 필수 쿠키 | 로그인 세션 유지, 장바구니 유지 | 보통 동의 없이도 허용 |
| 기능 쿠키 | 언어 설정, 다크모드 설정 기억 | 대체로 허용, 팝업에 포함되기도 함 |
| 분석 쿠키 | 방문자 수, 페이지 체류 시간 (Google Analytics 등) | 동의 필요 |
| 광고/마케팅 쿠키 | 본 상품을 다른 사이트에서 다시 광고(리타겟팅) | 동의 필요 |

### "수락"을 누르면 실제로 일어나는 일
1. 사이트가 자기 도메인 쿠키뿐 아니라 **제3자 쿠키(third-party cookie)**(광고/분석 회사)도 심을 수 있게 됨
2. 예: 쇼핑몰에 삽입된 광고 스크립트가 "이 사용자가 이 상품 페이지를 봤다"는 정보를 광고사 쪽에 쌓음
3. 이후 다른 사이트에 방문해도 같은 광고사의 슬롯이 있으면 제3자 쿠키로 동일 사용자를 인식해 관련 광고를 다시 노출 (리타겟팅 광고 원리)
4. "거부" 시 분석/광고용 쿠키는 심지 않고, 필수 쿠키만 사용

### 1st-party vs 3rd-party 쿠키
- **1st-party 쿠키**: 접속한 사이트 도메인이 직접 발급 (지금까지 다룬 세션 쿠키가 여기 해당)
- **3rd-party 쿠키**: 페이지에 삽입된 다른 도메인(광고사, 분석 툴)이 발급 → 크로스 사이트 추적의 핵심 수단. 최근 크롬 등 브라우저가 이를 점점 차단/제한하는 추세.

## 8. 세션의 다양한 용도 (트래킹 쿠키와의 차이)

### 쿠키 트래킹 vs 세션의 용도, 성격 차이
- 쿠키 트래킹: 사용자를 식별해서 **외부(광고/분석 회사)와 정보를 공유**하는 데 초점
- 세션: 서버 안에 갇힌 저장 공간이라 외부로 정보가 새는 개념이 아니라, **서비스 내부에서 사용자에 대해 뭘 임시로 기억할지**의 문제 (서비스 운영을 위한 임시 상태 관리)

### 세션의 실제 용도
1. **인증/인가 정보 보관** — 로그인 여부, 사용자 권한(관리자/일반 등)
2. **비로그인 사용자의 임시 데이터** — 장바구니처럼 로그인 전에도 담아두는 목록을 세션 단위로 임시 보관 (로그인 시 실제 계정 DB로 이관하기도 함)
3. **여러 단계로 나뉜 작업의 중간 상태 저장** — 다단계 회원가입 폼, 결제 프로세스(배송지 → 결제수단 → 최종확인) 등에서 이전 단계 입력값을 기억
4. **CSRF 방어용 토큰 저장** — 폼 렌더링 시 임의 토큰을 세션에 저장해두고, 제출 시 일치 여부 검증
5. **일시적인 서비스 정책 값** — 예: 로그인 실패 횟수를 세션에 잠깐 저장해 다음 시도 제한 (IP 단위로 별도 처리하는 경우도 많음)
6. **A/B 테스트 그룹 배정 값 기억** — 같은 사용자가 새로고침해도 같은 버전(A/B)을 계속 보게 배정 값을 세션에 저장

### 핵심 차이 한 줄 요약
- 쿠키의 추적: 사용자를 식별해 **외부와 정보 공유**
- 세션의 다양한 용도: 사용자를 식별해 **서비스 내부에서만 임시로 상태 기억**
- (다만 세션에 담긴 "최근 조회 상품" 등을 서비스 자체 개인화 추천에 쓰는 경우, 넓게 보면 서비스 내부용 행동 기록 역할과 겹치기도 함. 외부로 나가는 쿠키 트래킹과는 결이 다름)

## 10. 참고 사항 (수치 관련 주의)

- 본문에 나온 "쿠키 총 300개, 도메인당 20개, 개당 4KB" 수치는 RFC 6265에서 브라우저가 **최소한 지원해야 하는** 기준값. 실제 브라우저는 이보다 더 많이 허용하는 경우가 많으므로, 정확한 개수보다는 "쿠키는 용량과 개수에 제한이 있다"는 개념 위주로 답변하는 것이 안전.

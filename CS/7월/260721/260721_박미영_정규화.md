## 1. 정규화란?

- 데이터의 중복을 줄이고, 삽입 · 수정 · 삭제 과정에서 발생하는 이상 현상을 방지하기 위해 테이블을 구조적으로 분리하는 과정

## 2. 정규화를 하지 않으면 생기는 문제

| 학번 | 학생명 | 학과코드 | 학과명 | 과목코드 | 과목명 | 교수명 | 교수번호 | 성적 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 1001 | 김철수 | C01 | 컴퓨터공학과 | S01 | 데이터베이스 | 박교수 | 1 | A |
| 1001 | 김철수 | C01 | 컴퓨터공학과 | S02 | 운영체제 | 이교수 | 2 | B |
| 1002 | 이영희 | C01 | 컴퓨터공학과 | S01 | 데이터베이스 | 박교수 | 1 | A |
- 위 테이블에서는 `김철수`, `컴퓨터공학과`, `박교수` 같은 정보가 반복되고 있음.
- 여기서 발생할 수 있는 이상 현상 3가지
    - **삽입 이상**
        - 아직 수강생이 없는 새로운 과목을 등록하려고 해도 학번이 없어서 데이터를 넣기 어려움.
    - **수정 이상**
        - 컴퓨터공학과의 이름이 변경되면 관련된 모든 행을 수정해야 함.
        - 일부만 수정하면 데이터가 불일치함.
    - **삭제 이상**
        - 마지막 수강생의 수강 정보를 삭제했는데, 해당 과목과 교수 정보까지 사라질 수 있음.
        - 왜냐면, `이영희`는 데이터가 1개 밖에 없기 때문
- **여기서 알 수 있는 정규화가 필요한 이유 !!**
    - 이상 현상을 막기 위해서

## 3. 정규화 전에 알아야 할 개념

- **기본키**
    - 각 행을 유일하게 식별하는 속성
- **후보키**
    - 기본키가 될 수 있는 최소 속성 집합
- **함수의 종속성**
    - X의 값을 알면 Y의 값을 하나로 결정할 수 있을 때 `X -> Y` 라고 표현함.
    
    ```c
    학번 → 학생명, 학과코드
    학과코드 → 학과명
    과목코드 → 과목명, 교수번호
    교수번호 → 교수명
    학번, 과목코드 → 성적
    ```
    

## 4.. 제1정규형 : 원자값을 갖도록 만들기

- 하나의 컬럼에는 하나의 값만 저장해야 함.
- **잘못된 예시**
    
    
    | 학번 | 수강과목 |
    | --- | --- |
    | 1001 | 데이터베이스, 운영체제 |
    - `수강과목` 에 여러 값이 들어 있으므로 제1정규형을 만족하지 않음.
- **제1정규화 실행**
    
    
    | 학번 | 수강과목 |
    | --- | --- |
    | 1001 | 데이터베이스 |
    | 1001 | 운영체제 |
    - **“한 칸에 하나의 값”**

## 5. 제2정규형 : 부분 함수 종속 제거

- 제1정규형을 만족하면서, 기본키의 일부에만 종속되는 속성을 분리
- **기본키가 두 개 이상의 속성으로 구성된 복합키일 때 중요**

| 학번 | 학생명 | 학과코드 | 학과명 | 과목코드 | 과목명 | 교수명 | 교수번호 | 성적 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 1001 | 김철수 | C01 | 컴퓨터공학과 | S01 | 데이터베이스 | 박교수 | 1 | A |
| 1001 | 김철수 | C01 | 컴퓨터공학과 | S02 | 운영체제 | 이교수 | 2 | B |
| 1002 | 이영희 | C01 | 컴퓨터공학과 | S01 | 데이터베이스 | 박교수 | 1 | A |
- **기본키가 `(학번, 과목코드)`일 때, 위 테이블의 문제점**
    
    ```c
    학번 → 학생명
    과목코드 → 과목명
    학번, 과목코드 → 성적
    ```
    
    - `학생명`은 전체 기본키가 아니라 `학번`에만 의존
    - `과목명`은 `과목코드`에만 의존
    - **⇒ 부분 함수적 종속**
- **제2정규화 실행**
    
    ```c
    학생(학번, 학생명, 학과코드, 학과명)
    과목(과목코드, 과목명, 교수명)
    수강(학번, 과목코드, 성적)
    ```
    
    - **복합키의 일부에만 의존하는 속성을 제거하는 단계**
    - **완전 함수적 종속**

## 6. 제3정규형 : 이행 함수 종속 제거

- 제2정규형을 만족하면서, 기본키가 아닌 속성끼리의 종속 관계 제거

| 학번 | 학생명 | 학과코드 | 학과명 |
| --- | --- | --- | --- |
| 1001 | 김철수 | C01 | 컴퓨터공학과 |
| 1001 | 김철수 | C01 | 컴퓨터공학과 |
| 1002 | 이영희 | C01 | 컴퓨터공학과 |
- 위 테이블에서 종속 관계는 `학번 -> 학과코드`, `학과코드 -> 학과명`
- **결국 `학번 -> 학과코드 -> 학과명` 이라는 간접적인 종속 관계가 생김.**
- **제3정규화 실행**
    
    ```c
    학생(학번, 학생명, 학과코드)
    학과(학과코드, 학과명)
    ```
    
    - 위와 같이 분리할 수 있음.
    - 나머지도 마찬가지
    
    ```c
    과목(과목코드, 과목명, 교수번호)
    교수(교수번호, 교수명)
    ```
    
- **최종 테이블**
    
    ```c
    학생(학번, 학생명, 학과코드)
    학과(학과코드, 학과명)
    과목(과목코드, 과목명, 교수번호)
    교수(교수번호, 교수명)
    수강(학번, 과목코드, 성적)
    ```
    
    - **제3정규형은 기본키가 아닌 속성이 다른 일반 속성에 의존하지 않도록 하는 단계**

## 7. BCNF : 모든 결정자가 후보키

- 테이블에 존재하는 모든 함수적 종속 관계에서 결정자가 후보키여야 함.
- 함수적 종속 관계가 `X → Y`라면, `X`가 해당 테이블의 행을 유일하게 식별할 수 있는 후보키여야 한다는 뜻

| 학생 | 과목 | 교수 |
| --- | --- | --- |
| 김철수 | 데이터베이스 | 박교수 |
| 이영희 | 데이터베이스 | 박교수 |
| 김철수 | 운영체제 | 이교수 |
- 위 테이블에서 종속 관계는 `(학생, 과목) -> 교수`, `교수 -> 과목`
- 후보키는 `(학생, 과목)`, `(학생, 교수`) 두 가지가 될 수 있음.
- **그런데 `교수 -> 과목` 에서 결정자는 `교수`인데 교수만으로 학생을 알 수 없기 때문에 `교수`는 후보키가 아님. ⇒ BCNF 위반**
- **BCNF 실행**
    
    
    | 교수 | 과목 |
    | --- | --- |
    | 박교수 | 데이터베이스 |
    | 이교수 | 운영체제 |
    
    | 학생 | 교수 |
    | --- | --- |
    | 김철수 | 박교수 |
    | 이영희 | 박교수 |
    | 김철수 | 이교수 |
    - **BCNF는 모든 결정자가 후보키가 되도록 테이블을 분리하는 정규형**

## 8. 제4정규형 : 다치 종속 제거

- **다치 종속**
    - 하나의 속성에 대해 여러 개의 값이 독립적으로 존재하는 관계
    - `X →→ Y` : X 하나에 대해 Y가 여러 개 존재한다는 의미

| 개발자 | 기술 | 외국어 |
| --- | --- | --- |
| 김개발 | Java | 영어 |
| 김개발 | Java | 일본어 |
| 김개발 | Python | 영어 |
| 김개발 | Python | 일본어 |
- Java와 영어가 특별히 연결되어 있지 않음.
- 아래와 같은 다치 종속이 존재
    
    ```c
    개발자 →→ 기술
    개발자 →→ 외국어
    ```
    
    - **개발자 하나에 여러 기술과 여러 외국어가 각각 독립적으로 존재함.**
- **여기서 발생할 수 있는 문제**
    - 김개발이 새로운 기술을 React를 배운 경우, 외국어 수만큼 행을 추가해야 함.
    
    | 개발자 | 기술 | 외국어 |
    | --- | --- | --- |
    | 김개발 | React | 영어 |
    | 김개발 | React | 일본어 |
    - **불필요한 중복 발생**
- **제4정규화 실행**
    
    
    | 개발자 | 기술 |
    | --- | --- |
    | 김개발 | Java |
    | 김개발 | Python |
    
    | 개발자 | 외국어 |
    | --- | --- |
    | 김개발 | 영어 |
    | 김개발 | 일본어 |
    - **한 대상에 여러 목록이 서로 독립적으로 존재한다면 각각의 테이블로 분리**

## 9. 제5정규형 : 조인 종속 제거

- **조인 종속**
    - 하나의 테이블을 여러 개의 테이블로 분리한 뒤 다시 조인했을 때, 원래 테이블이 정확하게 복원되는 관계
- 조인 종속을 제거한다는 게 조인 종속을 없애는 게 아니라, 반복 저장된 복합 관계를 더 작은 관계로 분해한다는 의미

## 10. 반정규화(= 비정규화)

- 비정규화는 문맥에 따라 두 가지 의미로 사용됨.
    1. 정규형을 만족하지 않는 상태
    2. ‘반정규화’ 와 같은 의미로, 일부 교재나 실무에서 용어를 다르게 사용하는 것뿐 
- **반정규화란?**
    - 정규화된 테이블을 조회 성능 향상 등의 목적으로 의도적으로 다시 합치거나 중복 데이터를 저장하는 것
- 정규화를 하면 데이터 중복은 줄어들지만, 테이블이 여러 개로 나뉘어서 조회할 때 `JOIN`이 많아질 수 있음.

| 학번 | 학생명 | 학과코드 |
| --- | --- | --- |
| 1001 | 김철수 | C01 |

| 학과코드 | 학과명 |
| --- | --- |
| C01 | 컴퓨터공학과 |
- 위와 같은 경우에 학생의 학과명을 조회하려면 두 테이블을 조인해야 함.
    
    ```sql
    SELECT 학생.학생명, 학과.학과명
    FROM 학생
    JOIN 학과
    ON 학생.학과코드 = 학과.학과코드;
    ```
    
    - 이 조회가 자주 발생한다면 학생 테이블에 `학과명`을 중복 저장할 수 있음.
    
    | 학번 | 학생명 | 학과코드 | 학과명 |
    | --- | --- | --- | --- |
    | 1001 | 김철수 | C01 | 컴퓨터공학과 |
- **반정규화 방법**
    1. **테이블 병합**
        - 자주 조인하는 두 테이블을 하나로 합치는 방법
    2. **중복 컬럼 추가**
        - 다른 테이블에 있는 값을 중복 저장하는 방법
    3. **파생 컬럼 추가**
        - 계산할 수 있는 값을 미리 저장하는 방법
        - ex) 주문금액 x 수량으로 계산되는 총금액을 별도 저장
    4. **집계 테이블 추가**
        - 매번 집계하지 않고 결과를 미리 저장하는 방법
    5. **테이블 분할**
        - 하나의 큰 테이블을 조회 특성에 따라 나눔.
        - 수직 분할 : 컬럼을 나눔
        - 수평 분할 : 행을 조건별로 나눔
- **반정규화는 언제?**
    - 조인이 너무 많아 조회 속도가 느릴 때
    - 대량 데이터를 반복해서 집계할 때
    - 조회가 수정·삽입보다 훨씬 많을 때
    - 실제 성능 측정 결과 병목이 확인됐을 때

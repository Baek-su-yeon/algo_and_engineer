## 5번 — Java 배열 참조와 String 재할당

### 문제 핵심

Java에서 배열을 함수에 전달했을 때 배열 내부 값은 바뀔 수 있지만, `String s`를 함수 안에서 새 값으로 대입해도 원래 `main`의 `s`는 바뀌지 않는 이유를 묻는 문제이다.

```java
public class Main {
    public static void change(String[] data, String s){
        data[0] = s;
        s = "Z";
    }

    public static void main(String[] args) {
        String data[] = { "A" };
        String s = "B";
        change(data, s);
        System.out.print(data[0] + s);
    }
}
```

### 정답

```text
BB
```

### 해설

Java는 기본적으로 **값 전달(call by value)** 방식이다.  
하지만 배열 변수에는 배열 전체가 직접 들어 있는 것이 아니라, **배열 객체를 가리키는 참조값**이 들어 있다.

처음 상태는 다음과 같다.

```text
main의 data ──> 배열 객체 ["A"]
main의 s    ──> 문자열 객체 "B"
```

`change(data, s)`를 호출하면 `data`와 `s` 변수 자체가 넘어가는 것이 아니라, 각 변수 안에 들어 있던 값이 복사되어 전달된다.

여기서 복사되는 값은 다음과 같다.

```text
data 안에 있던 배열 참조값
s 안에 있던 문자열 참조값
```

따라서 함수 안에서는 다음과 같은 상태가 된다.

```text
main의 data   ──┐
                ├──> 같은 배열 객체 ["A"]
change의 data ─┘

main의 s      ──┐
                ├──> 문자열 객체 "B"
change의 s    ─┘
```

이 상태에서 다음 코드가 실행된다.

```java
data[0] = s;
```

이 코드는 `change` 함수 안의 `data`가 가리키는 배열 객체의 0번 칸을 바꾸는 것이다.  
그런데 `main의 data`와 `change의 data`가 같은 배열 객체를 가리키고 있으므로, 배열 내부 값이 실제로 바뀐다.

```text
배열 객체 ["A"] → ["B"]
```

반면 다음 코드는 다르다.

```java
s = "Z";
```

이 코드는 문자열 객체 자체를 바꾸는 것이 아니라, `change` 함수 안의 지역변수 `s`가 새 문자열 `"Z"`를 가리키도록 바꾸는 것이다.

```text
main의 s    ──> "B"
change의 s  ──> "Z"
```

따라서 `main`으로 돌아오면 다음 상태가 된다.

```text
data[0] = "B"
s       = "B"
```

그래서 출력은 다음과 같다.

```text
BB
```

### C/C++과 비교

C/C++에서도 배열을 함수 매개변수로 넘기면 배열 전체가 복사되는 것이 아니라, 포인터처럼 동작한다.

```c
void change(int arr[]) {
    arr[0] = 10;
}
```

위 코드는 사실상 다음과 유사하다.

```c
void change(int* arr) {
    arr[0] = 10;
}
```

따라서 C/C++에서도 배열 요소를 함수 안에서 바꾸면 원본 배열이 바뀔 수 있다.

단, Java 배열은 C/C++처럼 첫 번째 원소 주소에 포인터 연산을 하는 개념이라기보다, **배열 객체 전체를 가리키는 참조값**이라고 이해하는 것이 더 정확하다.

### 복습 포인트

```text
Java는 참조 전달이 아니라 참조값의 복사이다.
하지만 복사된 참조값이 같은 배열 객체를 가리키므로 배열 내부를 바꾸면 원본에도 보인다.

객체 내부 변경: 원본에 영향 있음
매개변수 변수 자체 재할당: 원본 변수에 영향 없음
```

## 6번 — 서브넷 마스크와 사용 가능한 호스트 수

### 문제 핵심

호스트 IP가 `223.13.234.132`, 서브넷 마스크가 `255.255.255.192`일 때 네트워크 주소와 사용 가능한 호스트 수를 구하는 문제이다.

### 정답

```text
네트워크 주소: 223.13.234.128
사용 가능한 호스트 수: 62개
```

### 해설

서브넷 마스크를 2진수로 바꾸면 다음과 같다.

```text
255.255.255.192

255 = 11111111
255 = 11111111
255 = 11111111
192 = 11000000
```

즉 전체 마스크는 다음과 같다.

```text
11111111.11111111.11111111.11000000
```

서브넷 마스크에서 규칙은 다음과 같다.

```text
1 = 네트워크 비트
0 = 호스트 비트
```

마지막 옥텟 `192`는 `11000000`이므로 다음과 같이 나눌 수 있다.

```text
11 000000
↑↑ ↑↑↑↑↑↑
네트워크 2비트 / 호스트 6비트
```

따라서 호스트 비트는 6개이다.

또는 `/26`으로 봐도 된다.

```text
255.255.255.192
= 11111111.11111111.11111111.11000000
= 1이 26개
= /26
```

IPv4는 전체가 32비트이므로 다음과 같이 계산할 수 있다.

```text
전체 32비트 - 네트워크 26비트 = 호스트 6비트
```

호스트 비트가 6개이면 만들 수 있는 전체 주소 수는 다음과 같다.

```text
2^6 = 64개
```

이 네트워크의 주소 범위는 다음과 같다.

```text
223.13.234.128  ← 네트워크 주소
223.13.234.129
223.13.234.130
...
223.13.234.190
223.13.234.191  ← 브로드캐스트 주소
```

여기서 맨 앞 주소와 맨 뒤 주소는 일반 호스트에게 할당할 수 없다.

- `223.13.234.128`: 네트워크 자체를 가리키는 주소
- `223.13.234.191`: 해당 네트워크의 모든 호스트에게 보내는 브로드캐스트 주소

따라서 실제로 할당 가능한 호스트 주소는 다음 범위이다.

```text
223.13.234.129 ~ 223.13.234.190
```

개수는 다음과 같다.

```text
전체 주소 64개 - 네트워크 주소 1개 - 브로드캐스트 주소 1개
= 64 - 2
= 62개
```

### 복습 포인트

```text
마스크의 1은 네트워크 자리
마스크의 0은 호스트 자리

호스트 수 = 2^(호스트 비트 수) - 2
```

## 9번 — Java 람다식과 예외 처리

### 문제 핵심

람다식으로 만든 함수를 `run` 메서드에 전달하고, `run`이 그 함수의 `apply(3)`을 실행했을 때 결과를 계산하는 문제이다.

```java
public class Main {
    static interface F {
        int apply(int x) throws Exception;
    }

    public static int run(F f) {
        try {
            return f.apply(3);
        } catch (Exception e) {
            return 7;
        }
    }

    public static void main(String[] args) {
        F f = (x) -> {
            if (x > 2) {
                throw new Exception();
            }
            return x * 2;
        };
        System.out.print(run(f) + run((int n) -> n + 9));
    }
}
```

### 정답

```text
19
```

### 해설

먼저 인터페이스 `F`를 보면 다음과 같다.

```java
static interface F {
    int apply(int x) throws Exception;
}
```

`F`는 `apply`라는 메서드 하나를 가진 인터페이스이다.  
즉 `F` 타입으로 사용하려면 다음 형태의 기능을 가져야 한다.

```java
int apply(int x) throws Exception;
```

이런 식으로 추상 메서드가 하나뿐인 인터페이스는 람다식으로 구현할 수 있다.

다음으로 `run` 메서드를 보면 다음과 같다.

```java
public static int run(F f) {
    try {
        return f.apply(3);
    } catch (Exception e) {
        return 7;
    }
}
```

이 메서드의 핵심은 다음 한 줄이다.

```java
return f.apply(3);
```

즉 `run`은 전달받은 `F` 타입 함수에 **항상 3을 넣어 실행**한다.

정상 실행되면 그 결과를 반환하고, 실행 중 예외가 발생하면 `catch`에서 7을 반환한다.

### 첫 번째 run(f)

`main`에서 첫 번째 람다는 다음과 같다.

```java
F f = (x) -> {
    if (x > 2) {
        throw new Exception();
    }
    return x * 2;
};
```

이 람다는 다음 함수처럼 이해하면 된다.

```text
apply(x):
    만약 x > 2이면 예외 발생
    아니면 x * 2 반환
```

`run(f)`를 호출하면 `run` 안에서 다음이 실행된다.

```java
f.apply(3)
```

따라서 첫 번째 람다에 `x = 3`이 들어간다.

```java
if (3 > 2) {
    throw new Exception();
}
```

`3 > 2`는 참이므로 예외가 발생한다.  
따라서 `return x * 2`는 실행되지 않는다.

예외가 발생했기 때문에 `run` 메서드의 `catch`로 이동한다.

```java
catch (Exception e) {
    return 7;
}
```

따라서 첫 번째 결과는 다음과 같다.

```text
run(f) = 7
```

### 두 번째 run((int n) -> n + 9)

두 번째 람다는 다음과 같다.

```java
(int n) -> n + 9
```

이것은 다음과 같은 의미이다.

```java
(int n) -> {
    return n + 9;
}
```

`run`은 여기에도 똑같이 `apply(3)`을 실행한다.

```text
n = 3
n + 9 = 12
```

예외가 발생하지 않으므로 그대로 12가 반환된다.

```text
run((int n) -> n + 9) = 12
```

최종 계산은 다음과 같다.

```text
run(f) + run((int n) -> n + 9)
= 7 + 12
= 19
```

### 람다식 기초 형태

Java 람다식의 기본 형태는 다음과 같다.

```java
(매개변수) -> { 실행문 }
```

예를 들어 다음 람다는:

```java
x -> x * 2
```

다음과 같은 의미이다.

```text
x를 받아서 x * 2를 반환한다.
```

여러 줄이면 중괄호와 return을 사용한다.

```java
x -> {
    int result = x * 2;
    return result;
}
```

Java에서 람다식은 아무 데나 단독으로 쓰는 것이 아니라, 보통 **추상 메서드가 하나뿐인 인터페이스**를 구현할 때 사용한다.

### 복습 포인트

```text
run은 항상 apply(3)을 실행한다.
첫 번째 람다는 3이 들어오면 예외 발생 → catch에서 7 반환
두 번째 람다는 3 + 9 → 12 반환
최종 결과 = 7 + 12 = 19
```

## 10번 — 인스턴스 메서드와 static 메서드의 선택 기준

### 문제 핵심

부모 타입 변수로 자식 객체를 가리킬 때, 인스턴스 메서드와 static 메서드가 각각 어떤 기준으로 실행되는지 묻는 문제이다.

```java
public class Main {
    public static class Parent {
        public int x(int i) { return i + 2; }
        public static String id() { return "P"; }
    }

    public static class Child extends Parent {
        public int x(int i) { return i + 3; }
        public String x(String s) { return s + "R"; }
        public static String id() { return "C"; }
    }

    public static void main(String[] args) {
        Parent ref = new Child();
        System.out.println(ref.x(2) + ref.id());
    }
}
```

### 정답

```text
5P
```

### 해설

다음 코드에서:

```java
Parent ref = new Child();
```

변수의 선언 타입은 `Parent`이지만 실제 객체는 `Child`이다.

```text
ref ──> Child 객체
```

### 인스턴스 메서드 x(2)

```java
ref.x(2)
```

`x(int i)`는 인스턴스 메서드이다.  
인스턴스 메서드는 **실제 객체 기준**으로 실행된다.

`ref`가 실제로 가리키는 객체는 `Child`이므로 `Child`의 `x(int i)`가 실행된다.

```java
public int x(int i) { return i + 3; }
```

따라서:

```text
ref.x(2) = 2 + 3 = 5
```

### static 메서드 id()

```java
ref.id()
```

`id()`는 static 메서드이다.  
static 메서드는 객체의 동작이 아니라 **클래스에 속한 기능**이다.

따라서 실제 객체 기준으로 동적으로 선택되지 않고, **참조 변수의 선언 타입 기준**으로 선택된다.

`ref`의 선언 타입은 `Parent`이므로 다음처럼 해석된다.

```java
Parent.id()
```

따라서 결과는 다음과 같다.

```text
ref.id() = "P"
```

최종 출력은 다음과 같다.

```text
5 + "P" = "5P"
```

### 왜 이런 개념이 생겼는가?

인스턴스 메서드는 객체의 실제 동작을 표현한다.  
따라서 부모 타입 변수로 자식 객체를 가리키더라도 실제 객체의 동작이 실행되어야 한다.

예를 들어:

```java
Animal a = new Dog();
a.sound();  // 실제 객체 Dog의 소리가 나야 함
```

이것이 다형성이다.

반면 static 메서드는 객체별 동작이 아니라 클래스 자체에 속한 기능이다.  
따라서 원래는 다음처럼 클래스 이름으로 호출하는 것이 적절하다.

```java
Parent.id();
Child.id();
```

Java에서는 `ref.id()`처럼 객체 변수로 static 메서드를 호출하는 것을 허용하지만, 실제로는 선언 타입 기준으로 선택된다.

### 복습 포인트

```text
인스턴스 메서드:
실제 객체 기준으로 실행된다.
오버라이딩이 적용된다.

static 메서드:
클래스에 속한 기능이다.
참조 변수의 선언 타입 기준으로 선택된다.
오버라이딩이 아니라 hiding이다.
```

## 13번 — 라운드로빈 평균 대기시간

### 문제 핵심

라운드로빈(Round Robin, RR) 스케줄링에서 시간 할당량이 주어졌을 때 평균 대기시간을 계산하는 문제이다.

조건은 다음과 같다.

```text
시간 할당량: 4ms
컨텍스트 스위칭 시간: 무시
```

| 프로세스 | 도착 시간 | 실행 시간 |
|---|---:|---:|
| P1 | 0 | 8 |
| P2 | 1 | 4 |
| P3 | 2 | 9 |
| P4 | 3 | 5 |

### 정답

```text
11.75ms
```

### 해설

라운드로빈은 한 프로세스가 CPU를 계속 독점하지 못하게, 정해진 시간만큼만 실행하고 아직 끝나지 않았으면 대기 큐의 뒤로 보내는 방식이다.

이 문제에서는 시간 할당량이 `4ms`이다.

```text
한 번 CPU를 잡으면 최대 4ms만 실행한다.
4ms 안에 끝나면 완료된다.
4ms를 써도 안 끝나면 남은 시간을 가지고 큐의 맨 뒤로 간다.
```

실행 순서는 다음과 같다.

```text
0~4    P1 실행: 8 → 4 남음
4~8    P2 실행: 4 → 0 완료
8~12   P3 실행: 9 → 5 남음
12~16  P4 실행: 5 → 1 남음
16~20  P1 실행: 4 → 0 완료
20~24  P3 실행: 5 → 1 남음
24~25  P4 실행: 1 → 0 완료
25~26  P3 실행: 1 → 0 완료
```

완료 시각은 다음과 같다.

| 프로세스 | 도착 시간 | 실행 시간 | 완료 시각 |
|---|---:|---:|---:|
| P1 | 0 | 8 | 20 |
| P2 | 1 | 4 | 8 |
| P3 | 2 | 9 | 26 |
| P4 | 3 | 5 | 25 |

대기시간은 다음 공식으로 구한다.

```text
대기시간 = 완료시각 - 도착시간 - 실행시간
```

이 공식의 의미는 다음과 같다.

```text
완료시각 - 도착시간 = 시스템 안에 머무른 전체 시간
전체 시간 - 실제 CPU 사용 시간 = 기다린 시간
```

각 프로세스별 대기시간은 다음과 같다.

```text
P1 = 20 - 0 - 8 = 12
P2 = 8 - 1 - 4 = 3
P3 = 26 - 2 - 9 = 15
P4 = 25 - 3 - 5 = 17
```

평균 대기시간은 다음과 같다.

```text
(12 + 3 + 15 + 17) / 4
= 47 / 4
= 11.75ms
```

### 복습 포인트

```text
RR 문제 풀이 순서

1. 시간 할당량을 확인한다.
2. 도착한 프로세스를 큐에 넣는다.
3. 큐 앞에서부터 최대 시간 할당량만큼 실행한다.
4. 끝나면 완료 시각을 기록한다.
5. 안 끝나면 남은 시간을 가지고 큐 맨 뒤로 보낸다.
6. 모든 프로세스 완료 후 대기시간을 계산한다.

대기시간 = 완료시각 - 도착시간 - 실행시간
평균 대기시간 = 각 대기시간 합 / 프로세스 수
```

## 15번 — Java 객체 참조와 배열 요소 교환

### 문제 핵심

객체 변수 `a`, `b`, `c`와 객체 배열 `arr`이 같은 객체들을 가리키고 있을 때, 배열 요소를 바꾸는 것과 객체 내부 값을 바꾸는 것이 어떤 차이를 가지는지 묻는 문제이다.

```java
public class Main {
    public static class BO {
        public int v;
        public BO(int v) {
            this.v = v;
        }
    }

    public static void main(String[] args) {
        BO a = new BO(1);
        BO b = new BO(2);
        BO c = new BO(3);
        BO[] arr = {a, b, c};
        BO t = arr[0];
        arr[0] = arr[2];
        arr[2] = t;
        arr[1].v = arr[0].v;
        System.out.println(a.v + "a" + b.v + "b" + c.v);
    }
}
```

### 정답

```text
1a3b3
```

### 해설

처음 객체 3개를 만든다.

```java
BO a = new BO(1);
BO b = new BO(2);
BO c = new BO(3);
```

상태는 다음과 같다.

```text
a ──> BO객체(v=1)
b ──> BO객체(v=2)
c ──> BO객체(v=3)
```

그다음 배열을 만든다.

```java
BO[] arr = {a, b, c};
```

배열에도 객체 자체가 들어가는 것이 아니라 객체 참조값이 들어간다.

```text
arr[0] ──> a 객체(v=1)
arr[1] ──> b 객체(v=2)
arr[2] ──> c 객체(v=3)
```

이제 다음 코드는 배열 안의 참조 위치를 바꾸는 코드이다.

```java
BO t = arr[0];
arr[0] = arr[2];
arr[2] = t;
```

실행 후 배열은 다음과 같이 바뀐다.

```text
arr[0] ──> c 객체(v=3)
arr[1] ──> b 객체(v=2)
arr[2] ──> a 객체(v=1)
```

즉 배열 기준으로 `[a, b, c]`였던 것이 `[c, b, a]`가 된 것이다.

하지만 여기서 중요한 점은 다음과 같다.

```text
a, b, c 변수 자체가 바뀐 것은 아니다.
배열 arr 안의 참조 순서만 바뀐 것이다.
```

따라서 여전히 다음 상태는 유지된다.

```text
a ──> BO객체(v=1)
b ──> BO객체(v=2)
c ──> BO객체(v=3)
```

이제 다음 코드가 실행된다.

```java
arr[1].v = arr[0].v;
```

현재 배열은 다음과 같다.

```text
arr[0] ──> c 객체(v=3)
arr[1] ──> b 객체(v=2)
arr[2] ──> a 객체(v=1)
```

따라서 다음 코드의 의미는:

```java
arr[1].v = arr[0].v;
```

다음과 같다.

```text
b 객체의 v = c 객체의 v
b.v = 3
```

그래서 최종 상태는 다음과 같다.

```text
a.v = 1
b.v = 3
c.v = 3
```

마지막 출력문은 다음과 같다.

```java
System.out.println(a.v + "a" + b.v + "b" + c.v);
```

계산하면 다음과 같다.

```text
1 + "a" + 3 + "b" + 3
= "1a3b3"
```

### 복습 포인트

```text
1. Java 객체 변수에는 객체 자체가 아니라 참조값이 들어간다.
2. 배열에도 객체 자체가 아니라 객체 참조값이 들어간다.
3. arr[0]과 arr[2]를 바꾸는 것은 배열 안의 참조 위치를 바꾸는 것이다.
4. a, b, c 변수 자체가 서로 바뀌는 것은 아니다.
5. arr[1].v = arr[0].v는 arr[1]이 가리키는 객체의 내부 값을 실제로 바꾸는 것이다.
```

## 18번 — 연결 리스트와 문자열 역순 출력

### 문제 핵심

문자열 `"BEST"`를 한 글자씩 읽어 연결 리스트에 저장하고, 그 리스트를 순서대로 출력했을 때 어떤 문자열이 출력되는지 묻는 문제이다.

```c
#include <stdio.h>
#include <stdlib.h>

struct node {
    char c;
    struct node* p;
};

struct node* func(char* s) {
    struct node* h = NULL, *n;
    while(*s) {
        n = malloc(sizeof(struct node));
        n->c = *s++;
        n->p = h;
        h = n;
    }
    return h;
}

int main() {
    struct node* n = func("BEST");
    while(n) {
        putchar(n->c);
        struct node* t = n;
        n = n->p;
        free(t);
    }
    return 0;
}
```

### 정답

```text
TSEB
```

### 구조체 의미

```c
struct node {
    char c;
    struct node* p;
};
```

이 구조체는 연결 리스트의 노드 하나를 의미한다.

```text
struct node 하나 =
문자 하나를 저장하는 c
다음 노드를 가리키는 포인터 p
```

즉 노드 하나는 다음과 같은 형태이다.

```text
[ 문자 | 다음 노드 주소 ]
```

### func 함수 흐름

```c
struct node* func(char* s) {
    struct node* h = NULL, *n;
    while(*s) {
        n = malloc(sizeof(struct node));
        n->c = *s++;
        n->p = h;
        h = n;
    }
    return h;
}
```

여기서 변수의 의미는 다음과 같다.

```text
s = 문자열을 한 글자씩 가리키는 포인터
h = 현재 연결 리스트의 맨 앞 노드, 즉 head
n = 새로 만든 노드
```

처음에는 리스트가 비어 있으므로:

```text
h = NULL
```

문자열 `"BEST"`는 메모리에 다음과 같이 저장된다.

```text
B E S T \0
```

`while(*s)`는 `s`가 가리키는 문자가 문자열 끝인 `\0`이 아닐 동안 반복한다.

### 핵심 코드

```c
n = malloc(sizeof(struct node));
n->c = *s++;
n->p = h;
h = n;
```

각 줄의 의미는 다음과 같다.

```text
n = malloc(sizeof(struct node));
→ 새 노드 하나를 만들 메모리 공간을 확보한다.

n->c = *s++;
→ 현재 문자를 새 노드에 저장하고 s를 다음 문자로 이동한다.

n->p = h;
→ 새 노드가 기존 head를 가리키게 한다.

h = n;
→ head를 새 노드로 바꾼다.
```

여기서 가장 중요한 두 줄은 다음이다.

```c
n->p = h;
h = n;
```

이 두 줄은 **새 노드를 기존 리스트의 앞에 삽입**한다.

따라서 문자열을 읽는 순서는 `B → E → S → T`이지만, 연결 리스트는 다음처럼 만들어진다.

```text
B 입력 → [B] → NULL
E 입력 → [E] → [B] → NULL
S 입력 → [S] → [E] → [B] → NULL
T 입력 → [T] → [S] → [E] → [B] → NULL
```

최종 리스트는 다음과 같다.

```text
h → [T] → [S] → [E] → [B] → NULL
```

즉 입력된 문자열이 거꾸로 저장된다.

### main 함수 출력 흐름

```c
while(n) {
    putchar(n->c);
    struct node* t = n;
    n = n->p;
    free(t);
}
```

`n`은 처음에 `func("BEST")`의 반환값, 즉 리스트의 head를 가리킨다.

```text
n → [T] → [S] → [E] → [B] → NULL
```

`putchar(n->c)`는 현재 노드의 문자를 출력한다.

출력 순서는 다음과 같다.

```text
T 출력
S 출력
E 출력
B 출력
```

따라서 최종 출력은 다음과 같다.

```text
TSEB
```

### malloc(sizeof(struct node))의 역할

```c
n = malloc(sizeof(struct node));
```

이 코드는 다음 의미이다.

```text
struct node 하나를 저장할 수 있는 메모리 공간을 새로 확보한다.
그리고 그 공간의 주소를 n에 저장한다.
```

`sizeof(struct node)`는 `struct node` 하나를 저장하는 데 필요한 메모리 크기이다.

즉:

```c
malloc(sizeof(struct node))
```

은 다음과 같이 이해하면 된다.

```text
node 하나가 들어갈 만큼의 메모리를 새로 빌려줘.
```

그 결과 `n`은 새 노드를 가리키게 된다.

```text
n ──> [ c를 저장할 칸 | p를 저장할 칸 ]
```

이후 `n->c`, `n->p`를 통해 그 공간에 값을 채운다.

```c
n->c = *s++;
n->p = h;
```

이 문제에서 `malloc`은 문자열의 각 글자를 저장할 노드를 하나씩 만드는 역할을 한다.

```text
B 노드 생성
E 노드 생성
S 노드 생성
T 노드 생성
```

### 왜 malloc을 쓰는가?

함수 안에서 일반 지역변수로 노드를 만들면 함수가 끝날 때 사라진다.

```c
struct node temp;
```

하지만 이 문제에서는 `func("BEST")`가 끝난 뒤에도 `main`에서 연결 리스트를 사용해야 한다.

```c
struct node* n = func("BEST");
```

따라서 함수가 끝나도 노드들이 살아 있어야 한다.  
그래서 `malloc`으로 메모리를 직접 확보한다.

```text
일반 지역변수: 함수가 끝나면 사라짐
malloc으로 만든 공간: free하기 전까지 계속 남아 있음
```

### free(t)의 역할

```c
free(t);
```

`malloc`으로 확보한 메모리는 자동으로 사라지지 않는다.  
다 쓴 뒤에는 직접 반납해야 한다.

`free(t)`는 다음 의미이다.

```text
t가 가리키는 메모리 공간을 이제 안 쓸 것이므로 반납한다.
```

출력 부분에서 다음 순서가 중요하다.

```c
struct node* t = n;
n = n->p;
free(t);
```

이 순서는 다음 의미이다.

```text
1. 현재 노드 주소를 t에 저장한다.
2. n을 다음 노드로 먼저 이동한다.
3. 이전 노드 t를 free한다.
```

만약 `free(n)`을 먼저 해버리면, 그 다음 노드 주소인 `n->p`를 안전하게 읽을 수 없다.  
이미 반납한 메모리를 다시 접근하는 꼴이 되기 때문이다.

따라서 반드시 다음 노드로 이동한 후 현재 노드를 해제해야 한다.

### 복습 포인트

```text
malloc(sizeof(struct node))
= struct node 하나가 들어갈 메모리 공간을 새로 확보한다.
= 새 노드 하나를 만든다.

free(t)
= malloc으로 확보했던 메모리를 더 이상 안 쓰겠다고 반납한다.
= 출력이 끝난 노드를 삭제한다.
```

18번의 핵심은 다음 한 줄이다.

```text
n->p = h;
h = n;

이 두 줄은 새 노드를 앞에 삽입하는 코드이다.
앞에 삽입하면 입력 순서가 뒤집힌다.
BEST → TSEB
```

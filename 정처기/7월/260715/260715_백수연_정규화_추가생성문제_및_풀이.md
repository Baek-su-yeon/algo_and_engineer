# 정규화 추가 생성 문제

> 2024년 1회 6번(정규형 판별) 오답 보강용 문제집
> 작성일: 2026-07-15

---

## 이 문제집 사용법

1. **Part 1(키 문제 10개)를 먼저 푼다.** 키를 못 찾으면 정규화는 아예 판별이 안 된다.
2. **Part 2(정규화 문제 20개)를 푼다.** 5NF는 제외했다(실기 출제 거의 없음).
3. 정답·해설은 **맨 뒤에 몰아놨다.** 먼저 풀고 나서 확인할 것.

### Part 2 답의 규칙 (중요)

Part 2는 전부 **"만족하는 최고 정규형"**을 묻는다. 답이 곧 **"바로 다음 단계를 위반했다"**는 뜻이다.

```
답이 "비정규형"  =  1NF 위반   (한 칸에 값 여러 개 / 반복 그룹)
답이 "1NF"       =  2NF 위반   (부분 함수 종속 있음)
답이 "2NF"       =  3NF 위반   (이행 함수 종속 있음)
답이 "3NF"       =  BCNF 위반  (후보키 아닌 결정자 있음)
답이 "BCNF"      =  4NF 위반   (다치 종속 있음)
```

### 판정 순서도

```
1. 한 칸에 값이 여러 개인가?          → YES: 비정규형
2. 기본키가 복합키인가?
   NO  → 부분종속 불가. 2NF 자동 통과. 3번으로
   YES → 키 일부만으로 정해지는 속성? → YES: 1NF
3. A → B → C 징검다리 있나?           → YES: 2NF
4. 결정자 중 후보키가 아닌 놈 있나?    → YES: 3NF
5. 독립적 다중값 2개가 조합 폭발?      → YES: BCNF
6. 다 통과                             → 완전 정규화
```

### 두문자 암기

```
원  -  부  -  이  -  결  -  다  -  조
원자값 부분종속 이행종속 결정자 다치종속 조인종속
1NF     2NF      3NF     BCNF    4NF      5NF
```

## 키-2

**학생(학번, 이름, 학과)** 릴레이션에서 **학번만이 유일성을 가진다**고 할 때, 다음 중 **슈퍼키를 모두 고르고**, 그중 **후보키**가 무엇인지 쓰시오. 또 그렇게 판단한 **기준**을 쓰시오.

**보기:**
```
ㄱ. (학번)
ㄴ. (이름)
ㄷ. (학번, 이름)
ㄹ. (이름, 학과)
ㅁ. (학번, 이름, 학과)
ㅂ. (학번, 학과)
```

## 키-6

```
학번 / 과목코드 / 성적 / 학생이름
```

또한 이 구분이 **3NF와 BCNF를 가르는 데 왜 중요한지** 한 줄로 쓰시오.

**기본키: 학번**
**대체키: 주민등록번호**

### 해설

**후보키 = 행을 유일하게 구분할 수 있는 최소 속성 집합.** 여기서 중요한 건 **"여러 개일 수 있다"**는 것이다.

- **학번**: 1001, 1002, 1003, 1004 — 전부 다름 → 유일성 O → 후보키 ⭕
- **주민등록번호**: 4개 전부 다름 → 후보키 ⭕
  - (주의: 990101-1과 990101-2는 뒷자리가 달라 서로 다른 값이다)
- **이름**: 동명이인 가능 → 후보키 ❌
- **학과**: 컴퓨터공학과가 2번 중복 → 후보키 ❌

**기본키 = 후보키 중에서 설계자가 "고른" 하나.**

> 여기가 네가 헷갈리는 지점이다.
> **기본키는 논리적으로 계산되어 나오는 게 아니라, 사람이 선택하는 것이다.**
> 후보키가 2개면 둘 다 기본키 자격이 있고, 그중 하나를 뽑는다.

학번을 고르는 이유: 짧고, 안 바뀌고, 개인정보가 아니다. 주민등록번호는 민감정보라 기본키로 부적절하다.

**대체키 = 후보키 중 기본키로 선택되지 못한 나머지.** 여기선 주민등록번호.

### 관계도

```
슈퍼키  ⊃  후보키  ⊃  기본키
                 └→ 나머지 = 대체키
```

## 키-2 정답

**슈퍼키: ㄱ, ㄷ, ㅁ, ㅂ**
**후보키: ㄱ (학번)**
**기준: 최소성**

### 해설

**슈퍼키 = 유일성만 있으면 됨 (군더더기 있어도 OK)**
**후보키 = 유일성 + 최소성 (군더더기 없어야 함)**

하나씩 보자:

| 보기 | 유일성 | 최소성 | 판정 |
| --- | --- | --- | --- |
| ㄱ. (학번) | O | O | **후보키** (슈퍼키이기도 함) |
| ㄴ. (이름) | X | - | 아무것도 아님 |
| ㄷ. (학번, 이름) | O | **X** | 슈퍼키 |
| ㄹ. (이름, 학과) | X | - | 아무것도 아님 |
| ㅁ. (학번, 이름, 학과) | O | **X** | 슈퍼키 |
| ㅂ. (학번, 학과) | O | **X** | 슈퍼키 |

**최소성이 왜 깨지나?**

`(학번, 이름)`을 보자. 학번만으로도 이미 행이 구분된다. **이름은 빼도 되는 군더더기**다. 뺄 수 있는 게 있으면 최소성 위반 → 후보키가 아니라 그냥 슈퍼키.

### 핵심 정리

```
슈퍼키:  "이거면 행을 구분할 수 있어"          (유일성)
후보키:  "이게 최소야. 하나라도 빼면 못 구분해"  (유일성 + 최소성)
기본키:  "후보키 중에 이걸 쓰기로 했어"         (선택)
```

> **모든 후보키는 슈퍼키다. 하지만 모든 슈퍼키가 후보키는 아니다.**

## 키-3 정답

**후보키: (학번, 과목코드)** — 복합 후보키

### 해설

단일 속성으로 안 되는 이유를 데이터로 확인하자:

- **학번만으로?** 1001이 2번(C01, C02) 나온다 → 행 구분 불가 ❌
- **과목코드만으로?** C01이 3번(1001, 1002, 1003) 나온다 → 행 구분 불가 ❌
- **(학번, 과목코드)?** (1001,C01), (1001,C02), (1002,C01)... 전부 유일 ⭕

**최소성 확인:** 학번을 빼면? 과목코드만 남는데 구분 안 됨. 과목코드를 빼면? 학번만 남는데 구분 안 됨.
→ **둘 다 필요하다 = 최소성 만족** ⭕

### 이게 중요한 이유

**후보키가 복합키인 순간, 2NF(부분종속) 검사가 필요해진다.**

```
기본키가 단일키  →  부분종속 불가능 → 2NF 자동 통과
기본키가 복합키  →  부분종속 의심해야 함
```

## 키-4 정답

**후보키: 사번, 이메일**

### 해설

데이터를 세로로 훑어서 중복이 있는지 본다.

| 속성 | 값들 | 중복? | 판정 |
| --- | --- | --- | --- |
| 사번 | E01, E02, E03, E04 | 없음 | **후보키** ⭕ |
| 사원명 | 김철수, 이영희, **김철수**, 박민수 | **있음** | ❌ |
| 이메일 | kim@, lee@, kim2@, park@ | 없음 | **후보키** ⭕ |
| 부서 | 개발팀, 개발팀, 인사팀, 인사팀 | 있음 | ❌ |

**사원명이 함정이다.** E01과 E03이 둘 다 "김철수"다. 동명이인이 실제로 데이터에 있으므로 후보키가 될 수 없다.

> **표에서 후보키 찾는 법: 그 열을 세로로 읽어서 같은 값이 두 번 나오면 탈락.**

## 키-5 정답

**기본키로 회원ID를 선택하는 이유 3가지:**

1. **NULL이 될 수 없어야 한다** — 휴대폰번호는 없는 회원이 있을 수 있다. 기본키는 **NULL 불가(개체 무결성)**.
2. **값이 자주 바뀌면 안 된다** — 휴대폰번호는 번호이동/해지로 바뀐다. 기본키가 바뀌면 이를 참조하는 모든 외래키를 다 고쳐야 한다.
3. **단순하고 짧아야 한다** — 인덱스 성능, 저장 공간에 유리하다.

**휴대폰번호 = 대체키(Alternate Key)**

### 해설

**기본키 선택 기준 (암기)**

```
① NOT NULL     — 반드시 값이 있어야 함 (개체 무결성)
② 유일성        — 중복 불가
③ 최소성        — 군더더기 없이
④ 불변성        — 잘 안 바뀌는 값
⑤ 단순성        — 짧고 간단하게
```

**개체 무결성 제약**: 기본키는 NULL일 수 없고 중복될 수 없다. 이건 시험에 그대로 나온다.

## 키-6 정답

**기본속성(Prime): 학번, 과목코드**
**비기본속성(Non-prime): 성적, 학생이름**

**중요한 이유:** 3NF에는 *"결정되는 쪽이 기본속성이면 통과"*라는 예외가 있지만 **BCNF에는 그 예외가 없기 때문.**

### 해설

정의는 단순하다.

```
기본속성   = 후보키에 "포함된" 속성
비기본속성 = 후보키에 "안 들어간" 속성
```

기본키가 (학번, 과목코드)이므로:

| 속성 | 후보키에 포함? | 분류 |
| --- | --- | --- |
| 학번 | O | **기본속성** |
| 과목코드 | O | **기본속성** |
| 성적 | X | 비기본속성 |
| 학생이름 | X | 비기본속성 |

### 왜 이게 3NF/BCNF를 가르나

```
3NF  : 모든 X → A 에서
       ① X가 후보키다              ← 또는
       ② A가 기본속성이다           ← 이 탈출구!

BCNF : 모든 X → A 에서
       ① X가 후보키다              ← 이것뿐
```

**②번 탈출구가 "A가 기본속성이냐"를 묻는다.** 그래서 기본속성/비기본속성 구분을 못 하면 3NF와 BCNF를 절대 구분할 수 없다.

2024년 1회 6번이 정확히 이 지점에서 갈렸다:
```
강사번호 → 강좌명
  강사번호: 후보키 아님 → ① 탈락
  강좌명:   기본속성 O  → ② 통과!   ∴ 3NF는 만족
                          BCNF엔 ②가 없음 → 위반
```

## 키-7 정답

**외래키: 학생 테이블의 학과코드** (학과 테이블의 기본키 학과코드를 참조)

**'가' 값은 들어갈 수 없다.** 학과 테이블에 존재하지 않는 값이므로 **참조 무결성 위반**이다.

### 해설

**외래키(Foreign Key) = 다른 릴레이션의 기본키를 참조하는 속성**

```
학생(학번, 이름, 학과코드)
                  └──────┐  참조
                         ↓
학과(학과코드, 학과명, 위치)
      └ 기본키
```

**참조 무결성 제약**: 외래키의 값은 반드시 **참조하는 테이블의 기본키에 존재하는 값**이거나 **NULL**이어야 한다.

- 학과 테이블에 D01, D02만 있음
- 학생.학과코드에 넣을 수 있는 값: **D01, D02, NULL** 뿐
- '가'는 학과 테이블에 없음 → **참조 무결성 위반** ❌

### 무결성 3종 비교 (시험 단골)

| 무결성 | 대상 | 규칙 |
| --- | --- | --- |
| **개체 무결성** | 기본키 | NULL 불가, 중복 불가 |
| **참조 무결성** | 외래키 | 참조 대상에 있는 값이거나 NULL |
| **도메인 무결성** | 일반 속성 | 정해진 자료형·범위를 지켜야 함 |

> **외래키는 NULL이 허용된다.** 기본키(NULL 불가)와 헷갈리지 말 것. 자주 나오는 함정이다.

## 키-8 정답

**후보키: 사번**

### 해설

**폐포(Closure) 구하는 법** — 후보키를 기계적으로 찾는 방법이다.

> **어떤 속성에서 출발해 화살표를 계속 따라갔을 때 전체 속성이 다 나오면, 그게 후보키다.**

**사번**부터 출발:

```
시작:        {사번}
사번 → 사원명:       {사번, 사원명}
사번 → 부서코드:     {사번, 사원명, 부서코드}
부서코드 → 부서명:   {사번, 사원명, 부서코드, 부서명}   ← 전체!
```

**사번⁺ = {사번, 사원명, 부서코드, 부서명} = 릴레이션 전체** → **사번은 후보키** ⭕

**다른 속성도 확인해보자:**

```
부서코드⁺ = {부서코드, 부서명}          → 전체 아님 ❌
사원명⁺   = {사원명}                    → 화살표 없음 ❌
부서명⁺   = {부서명}                    → 화살표 없음 ❌
```

**최소성:** 사번 단독으로 이미 전체가 나오므로 최소 ⭕

→ **후보키는 사번 하나뿐.**

### 요령

> **한 번도 화살표의 오른쪽에 안 나오는 속성은 반드시 후보키에 포함된다.**

사번은 어떤 화살표의 오른쪽에도 없다. 즉 **아무도 사번을 만들어주지 못한다.** 그러니 사번은 무조건 후보키에 들어가야 한다. 이 요령으로 출발점을 빨리 찾을 수 있다.

## 키-9 정답

**후보키: (학생, 과목), (학생, 교수)** — 2개

### 해설

**① (학생, 과목)**

```
(학생, 과목) → 교수   [주어진 FD]
∴ (학생,과목)⁺ = {학생, 과목, 교수} = 전체  ⭕
```
데이터 확인: (김민수,DB), (김민수,네트워크), (이지은,DB)... 전부 유일 ⭕
최소성: 학생만? 김민수가 2번 나옴 ❌ / 과목만? DB가 3번 ❌ → 둘 다 필요 ⭕

**→ 후보키** ⭕

**② (학생, 교수)** ← 이게 놓치기 쉽다

```
교수 → 과목           [주어진 FD]
∴ (학생, 교수) → 과목  (교수가 과목을 만들어줌)
∴ (학생,교수)⁺ = {학생, 교수, 과목} = 전체  ⭕
```
데이터 확인: (김민수,이교수), (김민수,박교수), (이지은,최교수)... 전부 유일 ⭕
최소성: 학생만? ❌ / 교수만? 이교수가 2번(김민수, 박서준) ❌ → 둘 다 필요 ⭕

**→ 후보키** ⭕

### 왜 이게 중요한가

**이 릴레이션이 바로 정규화-12번 문제다.**

```
교수 → 과목 에서
  교수는 후보키인가?  → (학생,교수)는 후보키지만 "교수" 단독은 아님
                       교수만으론 학생을 알 수 없다
  ∴ 교수는 후보키 아닌 결정자 → BCNF 위반
```

> **후보키를 다 못 찾으면 BCNF 판정을 틀린다.**
> 특히 **후보키끼리 속성이 겹치는 경우**(여기선 "학생"이 양쪽에 공통) — 이게 BCNF 문제의 전형적인 신호다.

## 키-10 정답

**사원명: 후보키가 될 수 없다.** — 지금 데이터엔 중복이 없지만, **동명이인이 언제든 생길 수 있기 때문.** 후보키는 데이터가 우연히 유일한 게 아니라 **업무 규칙상 항상 유일함이 보장**되어야 한다.

**이메일: 후보키가 될 수 없다.** — **E03의 이메일이 NULL**이다. 후보키는 NULL을 가질 수 없다.

### 해설

이 문제는 후보키의 **두 가지 함정**을 동시에 담고 있다.

**함정 ① — "지금 안 겹치니까 후보키"는 틀렸다**

| 사원명 |
| --- |
| 김철수 |
| 이영희 |
| 박민수 |
| 최지훈 |

지금은 4개가 다 다르다. 하지만 내일 "김철수"가 한 명 더 입사하면? **바로 깨진다.**

> **후보키 판단은 "현재 데이터"가 아니라 "업무 규칙상 영원히 유일한가"로 한다.**
> 이름, 전화번호, 주소 같은 건 원칙적으로 후보키가 아니다.

단, **시험 문제에서 표만 주고 "후보키를 찾으라"** 하면 → 주어진 데이터로 판단한다. 문제 의도를 봐야 한다. (키-4번이 그런 경우였고, 거기선 실제로 중복이 있어서 탈락했다)

**함정 ② — NULL이 있으면 후보키 불가**

박민수의 이메일이 비어있다. 후보키는 기본키가 될 자격이 있는 키인데, **기본키는 NULL 불가(개체 무결성)**다. 따라서 NULL이 있는 속성은 후보키가 될 수 없다.

### 정리

```
후보키의 자격
  ① 유일성      — 항상 중복 없음 (업무 규칙 기준)
  ② 최소성      — 군더더기 없음
  ③ NOT NULL    — 빈 값 없음
```

## 정규화-1 정답: **비정규형**

**이유:** 취득자격증 칸에 값이 여러 개 들어있다 → **원자값 위반 → 1NF 미만**

```
E01 → "정보처리기사, SQLD"   ← 값이 2개
E03 → "정보처리기사, ADsP, SQLD"  ← 값이 3개
```

**왜 문제인가:** `WHERE 취득자격증 = 'SQLD'` 로 검색하면 **아무도 안 잡힌다.** 그 칸의 값은 `'SQLD'`가 아니라 `'정보처리기사, SQLD'`라는 통짜 문자열이기 때문이다.

**해결 — 행으로 펼친다:**

| 사번 | 이름 | 취득자격증 |
| --- | --- | --- |
| E01 | 김민수 | 정보처리기사 |
| E01 | 김민수 | SQLD |
| E02 | 이지은 | 리눅스마스터 |
| E03 | 박서준 | 정보처리기사 |
| E03 | 박서준 | ADsP |
| E03 | 박서준 | SQLD |

(이름이 중복되기 시작하지만, 그건 2NF에서 해결할 문제다)

---

## 정규화-2 정답: **비정규형**

**이유:** **반복 그룹** → 1NF 위반

```
상품1 | 상품2 | 상품3   ← 같은 성격의 열에 번호를 붙여 옆으로 늘림
```

**칸마다 값은 하나씩이라 괜찮아 보이는 게 함정이다.** 이것도 1NF 위반이다.

**왜 문제인가:**

1. 검색이 지옥이 된다
   ```sql
   WHERE 상품1='마우스' OR 상품2='마우스' OR 상품3='마우스'
   ```
2. **4개 사는 고객이 나타나면 테이블 구조를 뜯어고쳐야 한다** (상품4 열 추가). 데이터가 늘었다고 스키마를 바꾼다는 건 설계 실패다.
3. NULL이 잔뜩 생긴다 (A002는 2칸이 빔)

**해결:**

| 주문번호 | 고객명 | 상품 |
| --- | --- | --- |
| A001 | 박서준 | 노트북 |
| A001 | 박서준 | 마우스 |
| A002 | 김민수 | 키보드 |
| A003 | 이지은 | 모니터 |
| A003 | 이지은 | 노트북 |
| A003 | 이지은 | 마우스 |

이제 10개를 사도 **행만 늘리면 된다. 구조는 그대로다.**

## 정규화-3 정답: **비정규형**

**이유:** 참여사원 칸에 "김철수, 이영희, 박민수" — 값이 3개 → 원자값 위반

**해결:**

| 프로젝트코드 | 프로젝트명 | 참여사원 |
| --- | --- | --- |
| PJ1 | 차세대시스템 | 김철수 |
| PJ1 | 차세대시스템 | 이영희 |
| PJ1 | 차세대시스템 | 박민수 |
| PJ2 | 앱개발 | 최지훈 |

> **1NF 3형제(1~3번) 정리:** 한 칸에 쉼표가 보이거나, 열 이름에 숫자가 붙어있으면 → **비정규형.** 답은 항상 "행으로 펼쳐라".

## 정규화-4 정답: **1NF** (2NF 위반)

**이유:** **부분 함수 종속** `과목코드 → 과목명`

### 판정 과정

**1단계 — 1NF?** 모든 칸에 값 하나씩 ⭕ 통과

**2단계 — 기본키가 복합키인가?** (학번, 과목코드) → **YES. 부분종속을 의심해야 한다.**

**3단계 — 키를 쪼개서 물어보기**

```
성적을 알려면?
  "1001의 C01 성적?" → A        ✅ 둘 다 필요
  "1001의 성적?"     → A? B?    ❌
  "C01의 성적?"      → A? C? B? ❌
  → 완전 함수 종속 ⭕ 문제없음

과목명을 알려면?
  "C01의 과목명?" → 데이터베이스   ✅ 학번이 필요 없다!
  → 키의 일부(과목코드)만으로 정해짐
```

### 🔴 부분 함수 종속: `과목코드 → 과목명`

데이터로 확인: C01 → 데이터베이스(3번 모두 일치), C02 → 운영체제(2번 모두 일치) ⭕

### 무엇이 문제인가

- **갱신 이상**: "데이터베이스"를 "DB개론"으로 바꾸려면 3개 행을 다 고쳐야 한다. 2개만 고치면 C01의 과목명이 두 가지가 되는 모순.
- **삽입 이상**: 아직 아무도 신청 안 한 신규 과목(C03, 컴파일러)을 **등록할 수 없다.** 학번이 기본키의 일부라 비워둘 수 없다.
- **삭제 이상**: C02 수강생이 전부 취소하면 → **"C02 = 운영체제"라는 정보가 증발한다.**

### 해결

```
수강(학번, 과목코드, 성적)    ← 키 전체가 필요한 것만
과목(과목코드, 과목명)        ← 부분종속된 놈을 분리
```

## 정규화-5 정답: **1NF** (2NF 위반)

**이유:** **부분 함수 종속** `상품코드 → 상품명, 단가`

```
기본키 = (주문번호, 상품코드)

주문수량:  (주문번호, 상품코드) → 주문수량    완전종속 ⭕
상품명:    상품코드 → 상품명                 부분종속 ❌
단가:      상품코드 → 단가                   부분종속 ❌
```

데이터 확인: P01 → 노트북/1200000 (O01, O02 모두 일치), P02 → 마우스/20000 (O01, O03 모두 일치) ⭕

**문제:** 노트북 단가가 바뀌면 P01이 들어간 모든 주문 행을 다 고쳐야 한다. 그리고 **한 번도 안 팔린 신상품은 등록조차 못 한다.**

**해결:**
```
주문상세(주문번호, 상품코드, 주문수량)
상품(상품코드, 상품명, 단가)
```

## 정규화-6 정답: **1NF** (2NF 위반)

**이유:** **부분 함수 종속** `사번 → 사원이름`

```
기본키 = (사번, 프로젝트번호)

참여개월:  (사번, 프로젝트번호) → 참여개월   완전종속 ⭕
           E01의 PJ1은 6개월, PJ2는 3개월. 둘 다 필요 ⭕
사원이름:  사번 → 사원이름                  부분종속 ❌
           "E01 이름?" → 김철수. 프로젝트번호 필요 없음
```

**"김철수"가 2번 중복 저장**되고 있다. 프로젝트 10개 참여하면 10번 적힌다.

**해결:**
```
사원프로젝트(사번, 프로젝트번호, 참여개월)
사원(사번, 사원이름)
```

## 정규화-7 정답: **1NF** (2NF 위반)

**이유:** **부분 함수 종속이 2개**

```
기본키 = (공급업체번호, 부품번호)

공급량:  (공급업체번호, 부품번호) → 공급량   완전종속 ⭕
업체명:  공급업체번호 → 업체명               부분종속 ❌  (왼쪽 키만)
부품명:  부품번호 → 부품명                   부분종속 ❌  (오른쪽 키만)
```

데이터 확인:
- S01 → 한국전자 (2행 일치), S02 → 대한상사 (2행 일치) ⭕
- P01 → 볼트 (2행 일치), P02 → 너트, P03 → 와셔 ⭕

**복합키의 양쪽에서 각각 부분종속이 발생한 경우다.** 이러면 테이블이 **3개**로 쪼개진다.

**해결:**
```
공급(공급업체번호, 부품번호, 공급량)
업체(공급업체번호, 업체명)
부품(부품번호, 부품명)
```

> **4~7번 정리:** 기본키가 복합키인데, **키의 한쪽만 봐도 답이 나오는 속성**이 있으면 → **부분종속 → 답은 1NF.**

## 정규화-8 정답: **2NF** (3NF 위반)

**이유:** **이행 함수 종속** `사번 → 부서코드 → 부서명`

### 판정 과정

**1단계 — 1NF?** ⭕

**2단계 — 기본키가 복합키인가?** **NO. 사번 단독이다.**

> **단일키 → 부분종속은 애초에 불가능 → 2NF 자동 통과** ⭕
> (쪼갤 "일부"가 없으니까)

**3단계 — 이행종속?**

```
사번 → 부서코드      E01이면 D1
부서코드 → 부서명    D1이면 개발팀
─────────────────────────────
∴ 사번 → 부서명      징검다리로 간접 성립 ❌
```

### 🔴 이행 함수 종속 발견

**중간에 낀 부서코드가 키가 아니라는 게 포인트다.** 키도 아닌 놈이 남의 다리 노릇을 하고 있다.

### 무엇이 문제인가

**"개발팀"이 2번, "인사팀"이 2번 중복.** 부서코드만 알면 부서명은 뻔한데 사원 수만큼 반복 저장된다.

- **갱신 이상**: 개발팀 → "플랫폼개발팀"으로 개명하면 소속 사원 300명 행을 다 고쳐야 한다.
- **삽입 이상**: **신설 부서 D3(기획팀)를 사원 없이 등록 못 한다.** 사번이 기본키인데 사원이 없다.
- **삭제 이상**: E03, E04가 퇴사하면 → **"D2 = 인사팀"이라는 정보가 통째로 사라진다.**

### 해결

```
사원(사번, 이름, 부서코드)    ← 부서코드는 외래키로 남김
부서(부서코드, 부서명)        ← 징검다리(B→C)를 독립 테이블로
```

## 정규화-9 정답: **2NF** (3NF 위반)

**이유:** **이행 함수 종속** `학번 → 우편번호 → 주소`

```
학번 → 우편번호      1001이면 06234
우편번호 → 주소      06234면 서울시 강남구
──────────────────────────────────
∴ 학번 → 주소        이행종속 ❌
```

데이터 확인: 06234 → 서울시 강남구 (1001, 1002 일치), 13494 → 성남시 분당구 (1003, 1004 일치) ⭕

기본키가 **학번 단독** → 2NF는 자동 통과. 하지만 **"서울시 강남구"가 2번 중복**되고 있다.

**해결:**
```
학생(학번, 이름, 우편번호)
주소(우편번호, 주소)
```

> 우편번호-주소는 이행종속의 **교과서적 예제**다. 기출에 자주 나오니 이 모양을 기억할 것.

## 정규화-10 정답: **2NF** (3NF 위반)

**이유:** **이행 함수 종속** `도서번호 → 출판사코드 → 출판사명, 출판사전화`

```
도서번호 → 출판사코드              B01이면 PB1
출판사코드 → 출판사명, 출판사전화   PB1이면 도우출판, 02-111-1111
─────────────────────────────────────────
∴ 도서번호 → 출판사명, 출판사전화   이행종속 ❌
```

기본키가 도서번호 단독 → 2NF 통과. 하지만 출판사 정보가 책 수만큼 반복된다.

**해결:**
```
도서(도서번호, 도서명, 출판사코드)
출판사(출판사코드, 출판사명, 출판사전화)
```

> **징검다리 오른쪽에 속성이 여러 개(출판사명, 전화)여도 이행종속은 이행종속이다.** 한꺼번에 딸려 나간다.

## 정규화-11 정답: **2NF** (3NF 위반) ← 함정 문제

**이유:** **이행 함수 종속** `(학번, 과목코드) → 점수 → 학점`

### 왜 함정인가

기본키가 **복합키 (학번, 과목코드)**라서 "복합키 = 2NF 검사"라고 반사적으로 생각하기 쉽다. 하지만 **부분종속은 없고 이행종속이 있다.**

> **복합키에서도 이행종속은 생긴다.** 복합키를 봤다고 2NF만 보고 끝내면 안 된다.

### 부분종속 검사 (없다)

```
학번 → 점수?      1001 → 95, 85      ❌ 안 정해짐
과목코드 → 점수?  C01 → 95, 75, 85   ❌ 안 정해짐
→ 부분종속 없음. 2NF 통과 ⭕
```

### 이행종속 검사 (있다)

```
(학번, 과목코드) → 점수      (1001,C01)이면 95
점수 → 학점                  95면 A
──────────────────────────────
∴ (학번, 과목코드) → 학점    이행종속 ❌
```

데이터로 `점수 → 학점` 확인:

| 점수 | 학점 |
| --- | --- |
| 95 | A, A |
| 85 | B, B |
| 75 | C |

모순 없이 일관됨 ⭕ → **함수 종속 성립**

### 무엇이 문제인가

**학점 기준이 바뀌면**(예: 95→A+) 95점인 모든 행을 다 고쳐야 한다. 그리고 **"90~100점은 A"라는 규칙 자체를 저장할 곳이 없다** — 해당 점수를 받은 학생이 있어야만 존재한다.

### 해결

```
성적(학번, 과목코드, 점수)
학점기준(점수, 학점)
```

---

## 정규화-12 정답: **3NF** (BCNF 위반)

**이유:** **교수는 결정자인데 후보키가 아니다.**

### 후보키 찾기 (키-9번과 동일)

```
(학생, 과목)  ⭕  후보키
(학생, 교수)  ⭕  후보키   ← 교수 → 과목 이므로 과목을 알아낼 수 있음
```

**후보키 2개가 "학생"을 공유하며 겹친다** — BCNF 문제의 전형적 신호다.

### 함수 종속 찾기

**① (학생, 과목) → 교수** — 후보키가 결정자 ⭕ 문제없음

**② 교수 → 과목** ← 범인

**교수 열을 세로로 훑어보자:**

| 교수 | 과목 |
| --- | --- |
| 이교수 | 데이터베이스, 데이터베이스 |
| 박교수 | 네트워크, 네트워크 |
| 최교수 | 데이터베이스 |

**이교수는 어디 나오든 무조건 데이터베이스. 박교수는 무조건 네트워크.** 모순되는 행이 하나도 없다 → **함수 종속 성립** ⭕

> **반대로 `과목 → 교수`는 성립하지 않는다.** 데이터베이스가 이교수이기도 최교수이기도 하다.

### 판정

**3NF는 통과** ⭕
```
교수 → 과목
  교수는 후보키인가?     → "교수" 단독으론 아님 → ① 탈락
  과목은 기본속성인가?   → 후보키 (학생,과목)의 일부 → ② 통과!
∴ 3NF 만족
```

**BCNF는 위반** ❌
```
BCNF엔 ②번 탈출구가 없다.
교수가 후보키가 아니므로 → 위반
```

### 🔴 정답: 3NF

### 해결

```
수강(학생, 교수)          ← 학생이 어떤 교수 수업을 듣는지
담당(교수, 과목)          ← 교수가 무슨 과목을 가르치는지
```

> **이게 2024년 1회 6번과 완전히 같은 구조다.** 6번의 (고객아이디, 강좌명, 강사번호)와 이 문제의 (학생, 과목, 교수)는 이름만 다르고 똑같다.

---

## 정규화-13 정답: **3NF** (BCNF 위반)

**이유:** **의사번호는 결정자인데 후보키가 아니다.**

### 함수 종속

```
(환자번호, 진료과) → 의사번호     후보키가 결정자 ⭕
의사번호 → 진료과                 ← 범인
```

**의사번호 열을 세로로 훑기:**

| 의사번호 | 진료과 |
| --- | --- |
| D01 | 내과, 내과 |
| D05 | 외과, 외과 |
| D02 | 내과 |

일관됨 ⭕ → `의사번호 → 진료과` 성립

**반대는?** `진료과 → 의사번호`: 내과 → D01, D02 → ❌ 성립 안 함

### 판정

```
후보키: (환자번호, 진료과), (환자번호, 의사번호)

의사번호 → 진료과
  의사번호는 후보키?  → 단독으론 아님. D01만 보고 환자를 못 정함 ❌
  진료과는 기본속성?  → O → 3NF 통과 ⭕
∴ 3NF 만족, BCNF 위반
```

### 해결
```
예약(환자번호, 의사번호)
의사(의사번호, 진료과)
```

---

## 정규화-14 정답: **3NF** (BCNF 위반)

**이유:** **강의는 결정자인데 후보키가 아니다.**

### 함수 종속

```
(요일시간, 강의실) → 강의    후보키가 결정자 ⭕
강의 → 강의실                ← 범인
```

**강의 열을 세로로 훑기:**

| 강의 | 강의실 |
| --- | --- |
| 데이터베이스 | 101호, 101호 |
| 네트워크 | 202호, 202호 |
| 운영체제 | 303호 |

일관됨 ⭕ → `강의 → 강의실` 성립 (각 강의는 지정 강의실이 있으므로)

**반대는?** `강의실 → 강의`: 101호 → 데이터베이스만... 어? 이것도 성립하는 것처럼 보인다.

> **여기가 이 문제의 미묘한 지점이다.** 주어진 데이터만 보면 `강의실 → 강의`도 성립해 보인다. 하지만 **업무 규칙은 "각 강의는 지정 강의실에서만 진행"**이지 "각 강의실은 한 강의만"이 아니다. 101호에서 다른 강의도 할 수 있다.
>
> **시험에서는 문제에 주어진 조건문(업무 규칙)이 데이터보다 우선한다.** 조건에 명시된 방향만 함수 종속으로 인정할 것.

### 판정

```
후보키: (요일시간, 강의실), (요일시간, 강의)

강의 → 강의실
  강의는 후보키?     → 단독으론 아님 ❌
  강의실은 기본속성?  → O → 3NF 통과 ⭕
∴ 3NF 만족, BCNF 위반
```

### 해결
```
시간표(요일시간, 강의)
강의실배정(강의, 강의실)
```

## 정규화-15 정답: **3NF** (BCNF 위반)

**이유:** **팀장번호는 결정자인데 후보키가 아니다.**

### 함수 종속

```
(사원번호, 업무영역) → 팀장번호    후보키가 결정자 ⭕
팀장번호 → 업무영역                ← 범인
```

**팀장번호 열을 세로로 훑기:**

| 팀장번호 | 업무영역 |
| --- | --- |
| L01 | 프론트엔드, 프론트엔드 |
| L02 | 백엔드, 백엔드 |
| L03 | 프론트엔드 |

일관됨 ⭕

**반대는?** `업무영역 → 팀장번호`: 프론트엔드 → L01, L03 → ❌

### 판정

```
후보키: (사원번호, 업무영역), (사원번호, 팀장번호)

팀장번호 → 업무영역
  팀장번호는 후보키?   → ❌ (L01만 보고 사원을 못 정함)
  업무영역은 기본속성? → O → 3NF 통과
∴ 3NF 만족, BCNF 위반
```

### 해결
```
배정(사원번호, 팀장번호)
팀장(팀장번호, 업무영역)
```

## 정규화-16 정답: **3NF** (BCNF 위반)

**후보키: (회원ID, 관심분야), (회원ID, 담당큐레이터)** — 2개

**이유:** **담당큐레이터는 결정자인데 후보키가 아니다.**

### 함수 종속

```
(회원ID, 관심분야) → 담당큐레이터   후보키가 결정자 ⭕
담당큐레이터 → 관심분야             ← 범인
```

**담당큐레이터 열을 세로로 훑기:**

| 담당큐레이터 | 관심분야 |
| --- | --- |
| C01 | 소설, 소설 |
| C03 | 과학, 과학 |
| C02 | 소설 |

일관됨 ⭕

**반대는?** `관심분야 → 담당큐레이터`: 소설 → C01, C02 → ❌

### 후보키 검증

**(회원ID, 담당큐레이터)가 왜 후보키인가?**
```
담당큐레이터 → 관심분야  이므로
(회원ID, 담당큐레이터) → 관심분야  가 성립
∴ (회원ID, 담당큐레이터)⁺ = 전체 속성  ⭕
```
데이터 확인: (M01,C01), (M01,C03), (M02,C02), (M02,C03), (M03,C01) — 전부 유일 ⭕

### 판정: 3NF

```
담당큐레이터 → 관심분야
  담당큐레이터는 후보키?  → 단독으론 아님 ❌
  관심분야는 기본속성?    → O → 3NF 통과
∴ BCNF 위반, 답은 3NF
```

> **12~16번 정리 — BCNF 위반의 공통 패턴:**
> ```
> 표가 3열이고
> (A, B) → C  이면서
> C → B       (C가 B를 결정)
> C는 후보키가 아님
> → 답은 무조건 3NF
> ```
> **찾는 법: 3번째 열(C)을 세로로 훑어서, 같은 값일 때 2번째 열(B)도 항상 같은지 본다.**

## 정규화-17 정답: **BCNF** (4NF 위반)

**이유:** **다치 종속** — 보유기술과 구사언어가 서로 독립인데 모든 조합이 저장됨

### 왜 BCNF까지는 통과하나

이 표에는 **함수 종속이 하나도 없다.**

```
사원번호 → 보유기술?  E01 → Java, Python   ❌ 하나로 안 정해짐
사원번호 → 구사언어?  E01 → 영어, 중국어    ❌
```

**화살표 하나짜리(→)가 없다 = 결정자가 없다 = BCNF가 위반할 게 없다** ⭕
(기본키는 세 속성 전부 = (사원번호, 보유기술, 구사언어))

### 🔴 다치 종속 (↠)

```
사원번호 ↠ 보유기술      E01 → {Java, Python}
사원번호 ↠ 구사언어      E01 → {영어, 중국어}
```

**Java를 안다고 영어를 하는 게 아니다. 둘은 완전히 무관하다.**

그런데 한 테이블에 있으니 **모든 조합(2×2=4행)**을 다 적어야 한다.

### 무엇이 문제인가

E01이 **기술 하나 추가**하면 → 언어가 2개니까 **2행을 추가**해야 한다.

```
기술 3개 × 언어 4개 = 12행 😱
```

**곱셈으로 폭발한다.**

### 해결 — 독립적인 두 덩어리를 분리

```
사원기술(사원번호, 보유기술)
사원언어(사원번호, 구사언어)
```

```
분리 전:  기술 3개 × 언어 4개 = 12행
분리 후:  기술 3행 + 언어 4행 =  7행
```

**곱셈이 덧셈으로 바뀐다. 이게 4NF의 효과다.**

---

## 정규화-18 정답: **BCNF** (4NF 위반)

**이유:** **다치 종속** — 수강과목과 동아리는 무관

```
학번 ↠ 수강과목     1001 → {데이터베이스, 네트워크}
학번 ↠ 동아리       1001 → {밴드부, 축구부}
```

**데이터베이스를 들어서 밴드부에 가입한 게 아니다.** 무관한 두 목록이라 2×2=4행이 강제로 만들어졌다.

함수 종속은 없다(`학번 → 수강과목`이 성립 안 함) → BCNF는 통과.

**해결:**
```
수강(학번, 수강과목)
동아리가입(학번, 동아리)
```

---

## 정규화-19 정답: **BCNF** (4NF 위반)

**이유:** **다치 종속** — 담당과정과 보유자격증은 무관

```
강사번호 ↠ 담당과정      T01 → {정보처리기사, 빅데이터분석}
강사번호 ↠ 보유자격증    T01 → {기술사, 기사}
```

빅데이터분석을 가르쳐서 기술사가 있는 게 아니다. 2×2=4행이 조합으로 생성됨.

**해결:**
```
강사과정(강사번호, 담당과정)
강사자격증(강사번호, 보유자격증)
```

> **17~19번 정리 — 4NF 위반의 신호:**
> - 표가 3열인데 **함수 종속이 하나도 없다**
> - 키 하나에 **서로 무관한 목록 2개**가 붙어있다
> - 그 조합이 **빠짐없이 전부(2×2=4행)** 적혀있다
> → **다치 종속 → 답은 BCNF**

---

## 정규화-20 정답: **BCNF 만족 (완전 정규화)** ← 함정

**이유:** **위반이 없다.** `배송기사 → 상품`이 성립할 것처럼 보이지만 **반례가 있다.**

### 왜 함정인가

표 모양이 12~16번(BCNF 위반)과 **똑같다.** 3열이고, 3번째 열이 2번째 열을 결정할 것처럼 생겼다. 반사적으로 "3NF!"라고 답하기 쉽다.

**하지만 데이터를 실제로 세로로 훑어야 한다.**

### `배송기사 → 상품` 검증

| 배송기사 | 상품 |
| --- | --- |
| K01 | 노트북, 노트북 | ⭕ 일관됨 |
| **K02** | **마우스, 노트북** | ❌ **반례!** |

**K02가 C01에게는 마우스를, C02에게는 노트북을 배송했다.**
→ K02 하나로 상품이 정해지지 않는다 → **함수 종속 성립 안 함** ❌

### 전체 판정

```
1. 1NF?         칸마다 값 하나 ⭕
2. 기본키?      (고객ID, 상품)
                (C01,노트북) (C01,마우스) (C02,노트북) (C03,노트북) 전부 유일 ⭕
3. 부분종속?    고객ID → 배송기사?  C01 → K01, K02  ❌ 없음
                상품 → 배송기사?    노트북 → K01, K02, K01  ❌ 없음
                → 2NF 통과 ⭕
4. 이행종속?    없음 → 3NF 통과 ⭕
5. 결정자?      (고객ID, 상품) 뿐이고, 이건 후보키다
                → BCNF 통과 ⭕
6. 다치종속?    배송기사는 키에 함수 종속됨. C01의 조합이 4개가 아니라 2개뿐
                (노트북-K01, 마우스-K02) → 독립적 다중값 아님
                → 4NF 통과 ⭕
```

### 🔴 정답: BCNF 만족 (더 분해할 필요 없음)

> **엄밀히는 다치종속도 조인종속도 없으므로 5NF까지 만족한다.**
> 다만 실기에서 "만족하는 최고 정규형"을 물으면 **BCNF**라고 답하면 된다.

### 이 문제가 주는 교훈

> **표 모양만 보고 답하지 마라. 반드시 열을 세로로 훑어 반례를 찾아라.**
>
> 함수 종속은 **"단 하나의 반례만 있어도 성립하지 않는다."**
> K02가 두 가지 상품을 배송한 행 하나 때문에 `배송기사 → 상품`이 통째로 깨진다.

## 정규형별 판별 요령 (한눈에)

| 정규형 | 신호 | 답 |
| --- | --- | --- |
| **비정규형** | 한 칸에 쉼표 / 열 이름에 숫자(상품1,2,3) | 1NF 위반 |
| **1NF** | 복합키인데 **키 한쪽만 봐도** 정해지는 속성 | 2NF 위반 (부분종속) |
| **2NF** | **A → B → C** 징검다리 (B가 키 아님) | 3NF 위반 (이행종속) |
| **3NF** | **키 아닌 열이 키의 일부인 열을 결정** | BCNF 위반 |
| **BCNF** | 함수종속 없음 + 무관한 목록 2개의 **모든 조합** | 4NF 위반 (다치종속) |

## 2NF vs 3NF vs BCNF — 화살표 모양으로 구분

```
부분종속 (→ 답 1NF)
   (A, B) → C  인데  A → C
   왼쪽이 키의 일부,  오른쪽이 비기본속성

이행종속 (→ 답 2NF)
   A → B → C
   B가 키가 아님,    C가 비기본속성

BCNF위반 (→ 답 3NF)
   (A, B) → C  인데  C → B
   왼쪽이 후보키 아님,  오른쪽이 기본속성!  ← 여기가 차이
```

> **부분종속과 BCNF위반의 결정적 차이:**
> ```
> 과목코드 → 과목명      과목코드=키의 일부,  과목명=비기본속성   → 부분종속 → 1NF
> 강사번호 → 강좌명      강사번호=키 아님,    강좌명=기본속성!    → BCNF위반 → 3NF
> ```
> **화살표 오른쪽이 기본키의 일부인지 아닌지**를 보면 갈린다.

## 키 개념 최종 정리

```
슈퍼키  ⊃  후보키  ⊃  기본키
                  └→ 나머지 = 대체키

슈퍼키:  유일성만                       (군더더기 OK)
후보키:  유일성 + 최소성 + NOT NULL     (여러 개 가능)
기본키:  후보키 중 설계자가 고른 하나     ← 계산이 아니라 "선택"
대체키:  기본키가 안 된 나머지 후보키
외래키:  다른 릴레이션의 기본키를 참조    ← NULL 허용됨
```

**무결성 3종**

| 무결성 | 대상 | 규칙 |
| --- | --- | --- |
| 개체 무결성 | 기본키 | NULL 불가, 중복 불가 |
| 참조 무결성 | 외래키 | 참조 대상에 있는 값이거나 NULL |
| 도메인 무결성 | 일반 속성 | 정해진 자료형·범위 준수 |

## 마지막 한 마디

**정규화는 단계가 높을수록 중복이 적다. 하지만 무조건 높은 게 정답은 아니다.**
테이블이 잘게 쪼개질수록 조인이 많아져 **조회 성능이 떨어진다.**
그래서 실무에선 일부러 되돌리기도 한다 → **반정규화(De-normalization)**. 이것도 시험에 나온다.

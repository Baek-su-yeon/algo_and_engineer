## 5번. Java 예외 처리와 주요 예외 정리

### 문제 핵심

```java
int a=5, b=0;
try {
    System.out.print(a/b);
} catch(ArithmeticException e) {
    System.out.print("출력1");
} catch(ArrayIndexOutOfBoundsException e) {
    System.out.print("출력2");
} catch(NumberFormatException e) {
    System.out.print("출력3");
} catch(Exception e) {
    System.out.print("출력4");
} finally {
    System.out.print("출력5");
}
```

`a / b`는 `5 / 0`이므로 Java에서 정수 0 나누기 예외인 `ArithmeticException`이 발생한다.

실행 흐름은 다음과 같다.

```text
try 안에서 a/b 실행
→ 5/0 발생
→ ArithmeticException 발생
→ catch(ArithmeticException e) 실행
→ "출력1" 출력
→ finally 실행
→ "출력5" 출력
```

정답:

```text
출력1출력5
```

### try-catch-finally 흐름

| 구문 | 역할 |
|---|---|
| `try` | 예외가 발생할 수 있는 코드를 작성 |
| `catch` | 특정 예외가 발생했을 때 처리 |
| `finally` | 예외 발생 여부와 관계없이 마지막에 실행 |

핵심은 `catch`는 위에서부터 검사하고, 맞는 예외를 만나면 그 catch 하나만 실행한다는 것이다. 이후 나머지 catch는 건너뛴다.

### Java 주요 예외 정리표
> 앵간하면 영어 직독하면 이해됨

| 예외 클래스 | 발생 상황 | 예시 | 시험/실무 포인트 |외우기 키워드|
|---|---|---|---|---|
| `ArithmeticException` | 산술 연산 오류 | `5 / 0` | 정수 0 나누기에서 자주 출제 |걍외워라|
| `ArrayIndexOutOfBoundsException` | 배열 인덱스 범위 초과 | `arr[5]` when length is 3 | 배열 문제 단골 |
| `StringIndexOutOfBoundsException` | 문자열 인덱스 범위 초과 | `"abc".charAt(5)` | 문자열 인덱스 문제 |
| `NullPointerException` | null 객체 사용 | `s.length()` when `s == null` | 실무에서 매우 자주 발생 |
| `NumberFormatException` | 숫자로 변환할 수 없는 문자열 | `Integer.parseInt("abc")` | 입력 변환 문제 |
| `ClassCastException` | 잘못된 형변환 | `(String)obj` when obj is Integer | 상속/다형성과 연결 |
| `IllegalArgumentException` | 잘못된 인자 전달 | 나이에 `-1` 전달 | 직접 예외를 던질 때 자주 사용 |불법(illegal)인자(argument)|
| `IllegalStateException` | 객체 상태가 부적절함 | 닫힌 객체를 다시 사용 | 인자는 맞는데 상태가 문제 |불법(illegal)상태(State)|
| `InputMismatchException` | 입력 타입 불일치 | `Scanner.nextInt()`에 문자 입력 | Scanner 입력 문제 |
| `IOException` | 입출력 오류 | 파일 읽기 실패 | Checked Exception |
| `FileNotFoundException` | 파일 없음 | 없는 파일 열기 | `IOException`의 하위 예외 |
| `SQLException` | DB 처리 오류 | SQL 실행 실패 | DB 연동 실무 |
| `InterruptedException` | 스레드 대기 중 인터럽트 | `Thread.sleep()` | 스레드 문제 |
| `Exception` | 일반 예외의 상위 클래스 | `catch(Exception e)` | 보통 마지막 catch에 둠 |
| `RuntimeException` | 실행 중 예외의 상위 클래스 | 여러 런타임 예외 포함 | Unchecked Exception의 대표 부모 |
| `Error` | JVM 수준의 심각한 오류 | `OutOfMemoryError` | 보통 직접 처리하지 않음 |

### 시험에서 특히 자주 나오는 예외

| 우선순위 | 예외 | 핵심 암기 |
|---|---|---|
| 1 | `ArithmeticException` | 정수 0 나누기 |
| 2 | `ArrayIndexOutOfBoundsException` | 배열 범위 초과 |
| 3 | `NullPointerException` | null 객체 사용 |
| 4 | `NumberFormatException` | 숫자로 바꿀 수 없는 문자열 |
| 5 | `StringIndexOutOfBoundsException` | 문자열 인덱스 초과 |
| 6 | `Exception` | 일반 예외의 상위 타입 |

### catch 순서 주의

부모 예외를 먼저 쓰면 자식 예외 catch가 실행될 수 없다.

잘못된 순서:

```java
try {
    int x = 5 / 0;
} catch (Exception e) {
    System.out.println("전체 예외");
} catch (ArithmeticException e) {
    System.out.println("산술 예외");
}
```

`ArithmeticException`은 `Exception`의 자식이므로, `Exception`이 먼저 모든 예외를 잡아버린다. 따라서 구체적인 예외를 위에, 포괄적인 예외를 아래에 둬야 한다.

```text
자식 예외 → 부모 예외
구체적인 예외 → 포괄적인 예외
```

### Checked Exception vs Unchecked Exception

| 구분 | 의미 | 예시 | 처리 강제 여부 |
|---|---|---|---|
| Checked Exception | 컴파일러가 예외 처리를 강제 | `IOException`, `SQLException`, `InterruptedException` | 반드시 `try-catch` 또는 `throws` 필요 |
| Unchecked Exception | 실행 중 발생 가능 | `ArithmeticException`, `NullPointerException`, `ArrayIndexOutOfBoundsException` | 처리 강제 안 함 |

시험 코드 문제에서는 대부분 Unchecked Exception이 자주 나온다.

## 6번. ARP / RARP와 보안 접근 통제 연계

### 문제 핵심

| 구분 | 의미 | 방향 |
|---|---|---|
| ARP | Address Resolution Protocol | IP 주소 → MAC 주소 |
| RARP | Reverse Address Resolution Protocol | MAC 주소 → IP 주소 |

ARP는 “이 IP 주소를 가진 장비의 MAC 주소가 뭐지?”를 알아내는 프로토콜이다.  
RARP는 이름 그대로 Reverse, 즉 반대 방향이다.

```text
ARP  = IP → MAC
RARP = MAC → IP
```

### 연계 기출: 접근 통제 방식

오늘 6번은 네트워크 프로토콜 문제지만, 보안 개념 정리에서 이전 회차의 접근 통제 문제와 같이 묶어두면 좋다.

#### 2023년 3회 7번 — 접근 통제 방식

| 방식 | 풀네임 | 핵심 |
|---|---|---|
| MAC | Mandatory Access Control | 강제적 접근 통제 |
| RBAC | Role-Based Access Control | 역할 기반 접근 통제 |
| DAC | Discretionary Access Control | 임의적 접근 통제 |

#### 2025년 3회 18번 — 접근 통제 방식

2025년 3회 18번도 같은 접근 통제 방식 문제였다.

| 설명 | 정답 |
|---|---|
| 중앙에서 보안 정책을 일괄 설정하고, 주체가 임의로 수정할 수 없음. 군사 기밀, 국가 보안 등 높은 보안 환경에서 사용 | MAC |
| 직무나 역할(Role)에 따라 접근 권한을 부여 | RBAC |
| 자원의 소유자가 접근 권한을 자유롭게 부여·회수 | DAC |

### MAC / RBAC / DAC 비교

| 방식 | 누가 권한을 정하나 | 기준 | 예시 | 보안성 느낌 |
|---|---|---|---|---|
| MAC | 중앙 관리자 / 시스템 | 보안 등급, 규칙 | 방산, 군사, 국가기밀 | 가장 엄격 |
| RBAC | 관리자 | 역할(Role) | 팀장, 부서장, 인사담당자 역할에 권한 부여 | 관리 편함 |
| DAC | 자원 소유자 | 소유자 판단 | 파일 소유자가 다른 사용자에게 읽기/쓰기 권한 부여 | 유연하지만 상대적으로 약함 |

암기:

```text
MAC = Mandatory = 강제 = 중앙에서 강제로 통제 = 방산/군사/국가기밀
RBAC = Role = 역할로 통제 = 팀장/관리자 역할
DAC = Discretionary = 재량/임의 = 소유자 마음대로 권한 부여
```

## ⭐ 8번. DB 기본 용어 정리 — 반복 출제 핵심

8번은 최근 몇 년 동안 조금씩 형태를 바꿔 반복 출제되는 DB 기본 용어 문제다. 특히 `Relation`, `Tuple`, `Attribute`, `Degree`, `Cardinality`, `Domain`, `Key`, `Integrity` 계열을 구분할 수 있어야 한다.

### 2025년 1회 8번 핵심

| 문제 설명 | 정답 |
|---|---|
| 릴레이션에서 속성의 개수 | Degree |
| 릴레이션에서 튜플의 개수 | Cardinality |
| 다른 릴레이션의 기본 키를 참조하는 속성 | Foreign Key |
| 특정 속성이 가질 수 있는 값의 유형·범위 | Domain |

### DB 기본 용어 정리표

| 용어 | 한글 표현 | 의미 | 쉽게 말하면 | 자주 나오는 포인트 |
|---|---|---|---|---|
| Relation | 릴레이션 | 관계형 DB에서 데이터를 저장하는 테이블 | 테이블 | 학생 테이블, 사원 테이블 |
| Tuple | 튜플 | 릴레이션의 행 | 한 줄 데이터 | 한 학생 정보, 한 사원 정보 |
| Attribute | 속성 | 릴레이션의 열 | 컬럼 | 이름, 학번, 나이, 부서 |
| Domain | 도메인 | 속성이 가질 수 있는 값의 범위나 유형 | 입력 가능한 값의 규칙 | 나이는 숫자, 성별은 남/여 등 |
| Degree | 차수 | 릴레이션의 속성 개수 | 열 개수 | 컬럼이 4개면 Degree = 4 |
| Cardinality | 카디널리티 | 릴레이션의 튜플 개수 | 행 개수 | 데이터가 5행이면 Cardinality = 5 |
| Schema | 스키마 | 데이터베이스의 구조나 설계 | 테이블 설계도 | 테이블명, 컬럼명, 자료형, 제약조건 |
| Instance | 인스턴스 | 어느 한 시점에 저장된 실제 데이터 | 현재 테이블 안의 값들 | 시간이 지나면 변할 수 있음 |
| Primary Key | 기본 키 | 튜플을 유일하게 식별하는 속성 | 대표 식별자 | 중복 불가, NULL 불가 |
| Candidate Key | 후보 키 | 기본 키가 될 수 있는 후보 | 유일 식별 가능한 키 후보 | 학번, 주민번호 등 |
| Alternate Key | 대체 키 | 후보 키 중 기본 키로 선택되지 않은 키 | 탈락한 후보 키 | 후보 키 - 기본 키 |
| Super Key | 슈퍼 키 | 튜플을 유일하게 식별할 수 있는 속성 집합 | 유일 식별 가능하면 됨 | 불필요한 속성이 포함될 수 있음 |
| Foreign Key | 외래 키 | 다른 테이블의 기본 키를 참조하는 속성 | 다른 테이블과 연결하는 키 | 참조 무결성과 연결 |
| Null | 널 | 값이 없거나 아직 모르는 상태 | 빈 값, 미정 값 | 0이나 공백과 다름 |
| Constraint | 제약조건 | 데이터의 정확성을 지키기 위한 규칙 | 입력 제한 규칙 | PK, FK, NOT NULL, UNIQUE 등 |
| Entity Integrity | 개체 무결성 | 기본 키는 NULL이나 중복이 될 수 없음 | PK 규칙 | 기본 키 문제에서 자주 나옴 |
| Referential Integrity | 참조 무결성 | 외래 키는 참조 대상에 실제 존재해야 함 | FK 규칙 | 없는 부서번호 참조 불가 |
| Domain Integrity | 도메인 무결성 | 속성 값은 정해진 도메인에 속해야 함 | 값 범위 규칙 | 나이에 문자 입력 불가 |
| View | 뷰 | 하나 이상의 테이블에서 만들어진 가상 테이블 | 조회용 가짜 테이블 | 실제 데이터 저장 X |
| Index | 인덱스 | 검색 속도를 높이기 위한 자료구조 | 책의 찾아보기 | 검색은 빠르지만 저장 공간 필요 |
| Normalization | 정규화 | 중복을 줄이고 이상현상을 방지하기 위해 테이블을 나누는 과정 | 깔끔하게 쪼개기 | 무결성 향상, 중복 감소 |
| Denormalization | 반정규화 | 성능 향상을 위해 일부러 중복을 허용하거나 테이블을 합침 | 빠르게 하려고 합치기 | 성능 향상, 무결성 저하 가능 |

### 특히 헷갈리는 묶음

| 헷갈리는 것 | 구분법 |
|---|---|
| Tuple vs Attribute | Tuple = 행, Attribute = 열 |
| Cardinality vs Degree | Cardinality = 행 개수, Degree = 열 개수 |
| Domain vs Attribute | Attribute = 컬럼 이름, Domain = 그 컬럼에 들어갈 수 있는 값의 범위 |
| Schema vs Instance | Schema = 구조, Instance = 실제 데이터 |
| Primary Key vs Foreign Key | Primary Key = 내 테이블의 대표 키, Foreign Key = 다른 테이블을 참조하는 키 |
| Candidate Key vs Alternate Key | Candidate Key = 후보 전체, Alternate Key = 후보 중 선택 안 된 것 |
| Super Key vs Candidate Key | Super Key = 유일 식별 가능하지만 군더더기 가능, Candidate Key = 최소 속성으로 유일 식별 |
| 정규화 vs 반정규화 | 정규화 = 중복 제거, 반정규화 = 성능 위해 중복 허용 |

### 암기용 한 줄 정리

```text
Relation = 테이블
Tuple = 행
Attribute = 열
Degree = 열 개수
Cardinality = 행 개수
Domain = 값의 범위
Schema = 구조
Instance = 실제 데이터
Primary Key = 대표 키
Foreign Key = 참조 키
```

## 9번. 브로드캐스트 의미 정리

9번은 서브넷 계산 자체보다 **브로드캐스트의 의미**를 놓치면 틀릴 수 있는 문제였다.

### 브로드캐스트란?

**브로드캐스트(Broadcast)**는 같은 네트워크 안에 있는 모든 장비에게 한 번에 보내는 통신이다.

일반 통신이 다음과 같다면,

```text
나 → 특정 한 명
```

브로드캐스트는 다음과 같다.

```text
나 → 같은 네트워크 안의 전원
```

따라서 문제에서 “브로드캐스팅으로 정보를 전달할 때 수신할 수 있는 IP”를 묻는다면, 결국 다음 의미다.

```text
내 IP와 같은 서브넷에 속한 IP를 모두 고르시오.
```

### 9번에 적용

문제의 PC:

```text
IP: 192.168.35.10
Subnet mask: 255.255.252.0
```

계산 결과 같은 네트워크 범위는 다음과 같다.

```text
192.168.32.0 ~ 192.168.35.255
```

보기의 IP들은 모두 이 범위에 속했다.

| 보기 | IP | 같은 네트워크 여부 |
|---|---|---|
| ㄱ | 192.168.34.1 | O |
| ㄴ | 192.168.32.19 | O |
| ㄷ | 192.168.35.200 | O |
| ㄹ | 192.168.33.138 | O |
| ㅁ | 192.168.35.50 | O |

정답:

```text
ㄱ, ㄴ, ㄷ, ㄹ, ㅁ
```

암기:

```text
브로드캐스트 = 같은 네트워크 전체에게 뿌리기
브로드캐스트 수신 가능 IP = 같은 서브넷에 속한 IP
```

## 10번. 아스키 코드 핵심 정리

10번은 문자 비교와 문자 뺄셈이 아스키 코드 값으로 계산된다는 점이 핵심이다.

### 반드시 외울 아스키 코드

| 문자 | 아스키 코드 |
|---|---:|
| ⭐`A` | 65 |
| `Z` | 90 |
| ⭐`a` | 97 |
| `z` | 122 |

### 암기 포인트

```text
대문자 A = 65
소문자 a = 97
대문자와 소문자 차이 = 32
```

같은 알파벳끼리는 다음 관계가 성립한다.

```text
'a' - 'A' = 32
'b' - 'B' = 32
'z' - 'Z' = 32
```

10번 문제에서는 다음 계산이 나왔다.

```text
Data[3] - Data[1]
= 'E' - 'A'
= 69 - 65
= 4
```

문자 비교도 내부적으로 숫자 비교다.

```text
'A' < 'B' < 'C' < ... < 'Z' < 'a' < ... < 'z'
```

## 12번. 결합도와 응집도 정리

### 2025년 1회 12번 정답

| 번호 | 설명 | 정답 |
|---|---|---|
| (1) 다른 모듈 내부에 있는 변수나 기능을 다른 모듈에서 사용하는 경우 | 내용 결합도 |
| (2) 배열, 오브젝트, 자료구조 등이 인터페이스로 전달되는 경우 | 스탬프 결합도 |
| (3) 모듈 밖에 선언된 전역 변수를 참조·갱신하는 경우 | 공통 결합도 |

### 결합도란?

**결합도(Coupling)**는 모듈과 모듈 사이가 얼마나 의존적인지를 의미한다.

```text
결합도 = 모듈 사이 관계
낮을수록 좋다
```

결합도가 높으면 한 모듈을 수정했을 때 다른 모듈까지 고장날 가능성이 커진다.

### 결합도 종류

좋음 → 나쁨 순서:

```text
자료 → 스탬프 → 제어 → 외부 → 공통 → 내용
```

나쁨 → 좋음 순서:

```text
내용 → 공통 → 외부 → 제어 → 스탬프 → 자료
```

| 결합도 | 의미 | 예시 | 좋고 나쁨 |
|---|---|---|---|
| 자료 결합도 | 모듈끼리 필요한 단순 데이터 값만 주고받음 | `func(name, age)` | 가장 좋음 |
| 스탬프 결합도 | 배열, 구조체, 객체 같은 자료구조 전체를 넘김 | `func(student)` | 비교적 좋음 |
| 제어 결합도 | 다른 모듈의 흐름을 제어하는 값을 넘김 | `func(true)`, `func(mode)` | 중간 |
| 외부 결합도 | 외부 장치, 외부 파일, 외부 형식 등을 여러 모듈이 함께 의존 | 공통 파일 형식, 외부 프로토콜 의존 | 나쁨 |
| 공통 결합도 | 여러 모듈이 전역 변수를 공유 | 전역 변수 `total`을 여러 함수가 수정 | 매우 나쁨 |
| 내용 결합도 | 한 모듈이 다른 모듈의 내부 변수나 내부 기능에 직접 접근 | 다른 모듈 내부 로직 직접 수정 | 최악 |

### 12번에 나온 세 가지

#### 내용 결합도

다른 모듈의 내부 변수나 내부 기능을 직접 사용하는 경우다.

```text
내용 결합도 = 다른 모듈의 내부를 직접 건드림 = 최악
```

#### 스탬프 결합도

함수나 모듈이 필요한 값만 받는 것이 아니라, 배열·구조체·객체 전체를 전달받는 경우다.

```text
스탬프 결합도 = 자료구조 전체를 넘김
```

예를 들어 이름만 필요하면 `name`만 받으면 되는데, `Student` 객체 전체를 받는 경우가 이에 가깝다.

#### 공통 결합도

전역 변수를 여러 모듈이 공유해서 사용하는 경우다.

```c
int total = 0;

void add() {
    total++;
}

void reset() {
    total = 0;
}
```

```text
공통 결합도 = 전역 변수 공유 = 위험
```

### 응집도란?

**응집도(Cohesion)**는 한 모듈 내부의 기능들이 얼마나 하나의 목적에 잘 모여 있는지를 의미한다.

```text
응집도 = 모듈 내부 관계
높을수록 좋다
```

응집도가 높으면 한 모듈이 하나의 목적에 집중하고 있다는 뜻이다.

### 응집도 종류

좋음 → 나쁨 순서:

```text
기능적 → 순차적 → 교환적/통신적 → 절차적 → 시간적 → 논리적 → 우연적
```

| 응집도 | 의미 | 예시 | 좋고 나쁨 |
|---|---|---|---|
| 기능적 응집도 | 모듈 안의 모든 기능이 하나의 목적을 위해 존재 | 로그인 처리 모듈 | 가장 좋음 |
| 순차적 응집도 | 앞 기능의 출력이 다음 기능의 입력이 됨 | 읽기 → 처리 → 저장 | 좋음 |
| 교환적/통신적 응집도 | 같은 데이터를 사용하는 기능들이 모임 | 같은 고객 데이터를 조회/수정 | 보통 이상 |
| 절차적 응집도 | 정해진 순서대로 실행되는 기능들이 모임 | 초기화 절차 모음 | 보통 |
| 시간적 응집도 | 같은 시점에 실행되는 기능들이 모임 | 프로그램 시작 시 실행 기능 모음 | 낮음 |
| 논리적 응집도 | 비슷한 성격의 기능들을 모음 | 입력 처리 기능 모음 | 낮음 |
| 우연적 응집도 | 관련 없는 기능들이 그냥 모임 | 잡다한 함수 모음 | 최악 |

### 결합도 vs 응집도

| 구분 | 결합도 | 응집도 |
|---|---|---|
| 영어 | Coupling | Cohesion |
| 보는 대상 | 모듈과 모듈 사이 | 모듈 내부 |
| 의미 | 다른 모듈과 얼마나 의존하는가 | 내부 기능들이 얼마나 한 목적에 모여 있는가 |
| 좋은 방향 | 낮을수록 좋음 | 높을수록 좋음 |
| 나쁜 상태 | 모듈끼리 너무 얽혀 있음 | 모듈 안에 잡다한 기능이 섞여 있음 |
| 좋은 설계 | 서로 독립적 | 하나의 목적에 집중 |

최종 암기:

```text
좋은 설계 = 낮은 결합도 + 높은 응집도
Low Coupling, High Cohesion
```

## 15번. 테스트 커버리지 종류 정리

15번 문제 자체는 문장 커버리지 문제였지만, 커버리지 종류가 많으므로 표로 구분해둔다.

### 커버리지 종류

| 커버리지 | 기준 | 쉽게 말하면 | 예시 포인트 |
|---|---|---|---|
| 문장 커버리지 | 프로그램의 모든 문장이 최소 1번 이상 실행되는가 | 코드 줄을 다 한 번씩 지나갔나 | 가장 기본적인 커버리지 |
| 결정 커버리지 | 모든 결정문의 결과가 True/False를 각각 최소 1번 만족하는가 | if문 전체 결과가 참/거짓 둘 다 나왔나 | 분기 커버리지라고도 함 |
| 조건 커버리지 | 결정문 안의 각 개별 조건식이 True/False를 각각 최소 1번 만족하는가 | `A && B`에서 A도 참/거짓, B도 참/거짓 나왔나 | 전체 if 결과는 놓칠 수 있음 |
| 조건/결정 커버리지 | 개별 조건식도 True/False, 전체 결정문도 True/False를 모두 만족하는가 | 조건 커버리지 + 결정 커버리지 | 둘 다 만족해야 함 |
| 변경 조건/결정 커버리지 | 각 개별 조건식이 전체 결정 결과에 독립적으로 영향을 주는가 | 조건 하나를 바꿨을 때 if 결과도 바뀌는지 확인 | MC/DC라고도 함 |
| 다중 조건 커버리지 | 결정문 안의 모든 개별 조건 조합을 전부 수행하는가 | `A && B`면 TT, TF, FT, FF 전부 확인 | 가장 꼼꼼하지만 테스트 수 많음 |
| 경로 커버리지 | 프로그램의 가능한 모든 실행 경로를 수행하는가 | 코드 흐름의 모든 길을 다 가봤나 | 반복문 있으면 현실적으로 어려움 |
| 분기 커버리지 | 각 분기 흐름을 최소 1번 이상 실행하는가 | if/else, switch case 각각 탔나 | 결정 커버리지와 거의 같은 의미로 자주 사용 |
| 루프 커버리지 | 반복문의 실행 횟수별 상황을 확인하는가 | 0회, 1회, 여러 회 반복을 테스트했나 | 반복문 테스트 기준 |

### 자주 헷갈리는 것

| 헷갈리는 커버리지 | 구분법 |
|---|---|
| 문장 vs 결정 | 문장은 “코드 줄 실행”, 결정은 “if 결과 참/거짓” |
| 결정 vs 조건 | 결정은 `if (A && B)` 전체, 조건은 A와 B 각각 |
| 조건 vs 조건/결정 | 조건은 개별 조건만, 조건/결정은 개별 조건 + 전체 결과 |
| 조건/결정 vs 변경 조건/결정 | 조건/결정은 단순히 둘 다 만족, 변경 조건/결정은 각 조건이 결과에 실제 영향을 줘야 함 |
| 다중 조건 vs 조건 | 조건은 각 조건의 T/F만 보면 됨, 다중 조건은 모든 조합을 봄 |
| 경로 vs 분기 | 분기는 갈림길 하나하나, 경로는 시작부터 끝까지 이어지는 전체 길 |

### 핵심 암기

```text
문장 = 모든 줄
결정 = if 전체 결과 T/F
조건 = 개별 조건 T/F
조건/결정 = 조건 + 결정
변경 조건/결정 = 각 조건이 결과를 바꾸는지
다중 조건 = 모든 조건 조합
경로 = 가능한 모든 실행 길
루프 = 반복 횟수별 확인
```

## ⭐ 16번. Java 재귀 / 분할 정복 / Math.max

### 코드

```java
public class Main {
    public static void main(String[] args) {
        int[] data = {3, 5, 8, 12, 17};
        System.out.println(func(data, 0, data.length - 1));
    }

    static int func(int[] a, int st, int end) {
        if (st >= end) return 0;
        int mid = (st + end) / 2;
        return a[mid] + Math.max(func(a, st, mid), func(a, mid + 1, end));
    }
}
```

정답:

```text
20
```

### 함수 의미

```java
func(a, st, end)
```

배열 `a`에서 `st`부터 `end`까지의 구간을 처리한다는 뜻이다.

처음 호출은 다음과 같다.

```text
func(data, 0, 4)
```

배열:

```text
인덱스:  0  1  2   3   4
값:      3  5  8  12  17
```

### 종료 조건

```java
if (st >= end) return 0;
```

`st == end`이면 원소가 하나 남은 상태지만, 이 문제에서는 그 값을 더하지 않고 `0`을 반환한다.

이게 가장 큰 함정이다.

```text
st == end이면 a[st]를 반환하는 것이 아니라 0을 반환한다.
```

### 핵심 return문

```java
return a[mid] + Math.max(func(a, st, mid), func(a, mid + 1, end));
```

말로 풀면 다음과 같다.

```text
현재 구간의 가운데 값 a[mid]를 더한다.
그리고 왼쪽 구간 결과와 오른쪽 구간 결과 중 더 큰 값을 추가로 더한다.
```

즉,

```text
현재 가운데 값 + max(왼쪽 재귀 결과, 오른쪽 재귀 결과)
```

양쪽을 모두 더하는 것이 아니라, 더 큰 쪽 하나만 고른다.

### 호출 추적

```text
func(0,4)
= a[2] + max(func(0,2), func(3,4))
= 8 + max(func(0,2), func(3,4))
```

왼쪽:

```text
func(0,2)
= a[1] + max(func(0,1), func(2,2))
= 5 + max(func(0,1), 0)

func(0,1)
= a[0] + max(func(0,0), func(1,1))
= 3 + max(0, 0)
= 3

func(0,2)
= 5 + max(3, 0)
= 8
```

오른쪽:

```text
func(3,4)
= a[3] + max(func(3,3), func(4,4))
= 12 + max(0, 0)
= 12
```

최종:

```text
func(0,4)
= 8 + max(8, 12)
= 20
```

### 호출 트리

```text
func(0,4)
├─ a[2] = 8
├─ func(0,2)
│  ├─ a[1] = 5
│  ├─ func(0,1)
│  │  ├─ a[0] = 3
│  │  ├─ func(0,0) = 0
│  │  └─ func(1,1) = 0
│  │  → 3
│  └─ func(2,2) = 0
│  → 5 + max(3,0) = 8
└─ func(3,4)
   ├─ a[3] = 12
   ├─ func(3,3) = 0
   └─ func(4,4) = 0
   → 12

최종: 8 + max(8,12) = 20
```

### 함정 정리

| 함정 | 설명 |
|---|---|
| 모든 값을 더하는 문제가 아님 | `Math.max` 때문에 왼쪽/오른쪽 중 큰 쪽만 선택 |
| 마지막 원소 17이 더해지지 않음 | `func(4,4)`는 `st >= end`라서 0 반환 |
| `st == end` 주의 | 원소 하나가 남아도 그 값을 반환하지 않음 |

핵심:

```text
현재 구간의 가운데 값을 더하고, 왼쪽/오른쪽 재귀 결과 중 큰 쪽만 선택하는 재귀 문제
```

## 18번. 연결 리스트 — 특정 노드 맨 앞으로 이동

### 코드 핵심

```c
typedef struct Data {
    int value;
    struct Data *next;
} Data;
```

노드 하나는 값과 다음 노드를 가리키는 포인터를 가진다.

```text
[value | next]
```

### insert 함수

```c
Data* insert(Data* head, int value) {
    Data* new_node = (Data*)malloc(sizeof(Data));
    new_node->value = value;
    new_node->next = head;
    return new_node;
}
```

핵심은 다음 줄이다.

```c
new_node->next = head;
```

새 노드를 기존 head 앞에 붙인다. 즉, insert는 뒤 삽입이 아니라 앞 삽입이다.

### main의 삽입 결과

```c
Data *head = NULL, *curr;
for (int i = 1; i <= 5; i++)
    head = insert(head, i);
```

각 단계:

```text
1 삽입 → [1]
2 삽입 → [2] → [1]
3 삽입 → [3] → [2] → [1]
4 삽입 → [4] → [3] → [2] → [1]
5 삽입 → [5] → [4] → [3] → [2] → [1]
```

따라서 삽입 후 리스트:

```text
head
 ↓
[5] → [4] → [3] → [2] → [1] → NULL
```

### reconnect 함수 목적

```c
head = reconnect(head, 3);
```

값이 3인 노드를 찾아서 맨 앞으로 옮긴다.

초기 상태:

```text
[5] → [4] → [3] → [2] → [1] → NULL
```

3을 맨 앞으로 옮기면:

```text
[3] → [5] → [4] → [2] → [1] → NULL
```

### prev와 curr

| 변수 | 의미 |
|---|---|
| `curr` | 현재 검사 중인 노드 |
| `prev` | 현재 노드 바로 앞 노드 |

값 3을 찾았을 때 상태:

```text
[5] → [4] → [3] → [2] → [1]
        ↑     ↑
       prev  curr
```

### 노드 이동 핵심 3줄

```c
prev->next = curr->next;
curr->next = head;
head = curr;
```

각 줄의 의미:

| 코드 | 의미 |
|---|---|
| `prev->next = curr->next;` | curr 노드를 기존 자리에서 뺀다 |
| `curr->next = head;` | curr 뒤에 기존 head를 붙인다 |
| `head = curr;` | curr를 새 head로 만든다 |

과정:

```text
초기:
5 → 4 → 3 → 2 → 1

3을 기존 자리에서 제거:
5 → 4 → 2 → 1
3

3을 맨 앞에 붙임:
3 → 5 → 4 → 2 → 1
```

마지막 출력:

```text
35421
```

### 시험에서 푸는 순서

```text
1. 구조체가 어떤 노드 모양인지 본다.
2. insert가 앞 삽입인지 뒤 삽입인지 본다.
3. main에서 리스트 초기 상태를 그린다.
4. prev, curr가 어디를 가리키는지 표시한다.
5. next가 바뀌는 줄만 집중해서 연결을 다시 그린다.
6. head부터 출력한다.
```

핵심:

```text
prev->next = curr->next;  // curr를 기존 자리에서 뺌
curr->next = head;        // curr 뒤에 기존 head를 붙임
head = curr;              // curr를 새 head로 만듦
```

## 19번. 16진수 / 2진수 / 10진수 변환 정리

19번은 코드 자체보다 진법 변환 감각이 중요했다. 특히 `0xA5` 같은 16진수와 비트 AND 연산이 함께 나온다.

### 기본 진법

| 진법 | 의미 | 사용 숫자 |
|---|---|---|
| 2진수 | 0과 1만 사용 | `0, 1` |
| 10진수 | 평소 쓰는 숫자 | `0~9` |
| 16진수 | 16개 기호 사용 | `0~9, A~F` |

### 16진수 문자 값

| 16진수 | 10진수 |
|---|---:|
| A | 10 |
| B | 11 |
| C | 12 |
| D | 13 |
| E | 14 |
| F | 15 |

### 16진수 ↔ 2진수

가장 중요하다.

```text
16진수 한 자리 = 2진수 4자리
```

| 16진수 | 2진수 |
|---|---|
| 0 | 0000 |
| 1 | 0001 |
| 2 | 0010 |
| 3 | 0011 |
| 4 | 0100 |
| 5 | 0101 |
| 6 | 0110 |
| 7 | 0111 |
| 8 | 1000 |
| 9 | 1001 |
| A | 1010 |
| B | 1011 |
| C | 1100 |
| D | 1101 |
| E | 1110 |
| F | 1111 |

예시:

```text
2F₁₆
= 2  F
= 0010 1111₂
```

반대로 2진수를 16진수로 바꿀 때는 오른쪽부터 4자리씩 끊는다.

```text
101111₂
= 0010 1111₂
= 2F₁₆
```

### 2진수 → 10진수

오른쪽부터 `2^0, 2^1, 2^2, ...` 자리값을 가진다.

예시:

```text
1011₂
= 1×2³ + 0×2² + 1×2¹ + 1×2⁰
= 8 + 0 + 2 + 1
= 11₁₀
```

### 10진수 → 2진수

2로 계속 나누고 나머지를 아래에서 위로 읽는다.

예시: 13

```text
13 ÷ 2 = 6 ... 1
6 ÷ 2 = 3 ... 0
3 ÷ 2 = 1 ... 1
1 ÷ 2 = 0 ... 1
```

아래에서 위로 읽으면:

```text
13₁₀ = 1101₂
```

### 16진수 → 10진수

오른쪽부터 `16^0, 16^1, 16^2, ...` 자리값을 가진다.

예시:

```text
2F₁₆
= 2×16¹ + F×16⁰
= 2×16 + 15×1
= 32 + 15
= 47₁₀
```

### 10진수 → 16진수

16으로 계속 나누고 나머지를 아래에서 위로 읽는다.

예시: 47

```text
47 ÷ 16 = 2 ... 15
2 ÷ 16 = 0 ... 2
```

15는 16진수에서 F이므로:

```text
47₁₀ = 2F₁₆
```

### 19번 비트 AND 계산 방식

`0xA5`를 2진수로 바꾸면:

```text
0xA5 = 1010 0101₂
```

AND 연산은 두 비트가 모두 1일 때만 1이다.

| 계산 | 2진수 결과 | 16진수 결과 | 10진수 |
|---|---|---|---:|
| `0xA0 & 0xA5` | `1010 0000` | `0xA0` | 160 |
| `0xA5 & 0xA5` | `1010 0101` | `0xA5` | 165 |
| `0xDB & 0xA5` | `1000 0001` | `0x81` | 129 |
| `0xED & 0xA5` | `1010 0101` | `0xA5` | 165 |
| `0x81 & 0xA5` | `1000 0001` | `0x81` | 129 |

문제에서는 다음 값들을 더한다.

```text
160 + 165 + 129 + 160 + 165 + 129 = 908
```

정답:

```text
908
```

### 진법 변환 핵심 암기

```text
16진수 한 자리 = 2진수 4자리
A=10, B=11, C=12, D=13, E=14, F=15
2진수 → 10진수 = 1인 자리의 2의 거듭제곱을 더한다
10진수 → 2진수 = 2로 나누고 나머지를 거꾸로 읽는다
16진수 → 10진수 = 각 자리 × 16의 거듭제곱
10진수 → 16진수 = 16으로 나누고 나머지를 거꾸로 읽는다
```

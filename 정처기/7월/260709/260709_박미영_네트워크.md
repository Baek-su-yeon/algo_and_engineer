# 4. 네트워크

### OSI 참조 모델 (7계층)

- 데이터가 내 컴퓨터에서 상대 컴퓨터까지 갈 때 거치는 7단계
- ex) 카톡으로 “안녕”을 보낼 때
    
    ```c
    내가 카톡에 "안녕" 입력
    ↓
    컴퓨터가 이걸 네트워크로 보낼 수 있게 포장
    ↓
    주소 붙이고
    ↓
    전송 규칙 적용하고
    ↓
    전기/무선 신호로 바꿔서 보냄
    ↓
    상대 컴퓨터가 다시 해석해서 "안녕"으로 보여줌
    ```
    
- **응표세전네데물**
    - 응용, 표현, 세션, 전송, 네트워크, 데이터링크, 물리

| 응용 계층 | 사용자가 직접 쓰는 서비스 | HTTP, FTP, SMTP, DNS |
| --- | --- | --- |
| 표현 계층 | 데이터를 보기 좋게/안전하게 변환 | 암호화, 압축, 코드 변환 |
| 세션 계층 | 통신 연결을 만들고 유지하고 종료 | 연결 생성, 유지, 종료 |
| 전송 계층 | 상대 프로그램까지 데이터가 잘 가게 함 | TCP, UDP, 오류/흐름 제어 |
| 네트워크 계층 | 목적지까지 갈 길을 찾음 | IP, 라우팅, 경로 설정 |
| 데이터 링크 계층 | 바로 옆 장치까지 프레임 전송 | MAC 주소, 프레임, HDLC |
| 물리 계층 | 0과 1을 전기/무선 신호로 바꿈 | 전기적 신호, 케이블 |

### TCP/IP 4계층

- OSI 7계층은 이론적으로 7단계고, TCP/IP 4계층은 실제 인터넷에서 많이 쓰는 구조
    
    
    | OSI 7계층 | TCP/IP 4계층 |
    | --- | --- |
    | 응용 + 표현 + 세션 | **응용 계층** |
    | 전송 | **전송 계층** |
    | 네트워크 | **인터넷 계층** |
    | 데이터 링크 + 물리 | **네트워크 액세스 계층** |
- **계층별 대표 프로토콜**
    - 각 계층에서 통신을 처리할 때 사용하는 대표적인 규칙들
    
    | TCP/IP 계층 | 대표 프로토콜 |
    | --- | --- |
    | **응용 계층** | Telnet, FTP, HTTP, SMTP, DNS, SNMP, DHCP |
    | **전송 계층** | TCP, UDP, RTCP |
    | **인터넷 계층** | IP, ICMP, ARP, RARP |
    | **네트워크 액세스 계층** | HDLC, PPP, LLC |

### 데이터 링크 계층 프로토콜

| HDLC (High Level Data Link Control) | 대표적인 데이터링크 프로토콜 / 비트 동기 방식 |
| --- | --- |
| BSC (Binary Synchronous Communication) | 프레임에 전송 제어 문자 삽입 |
| LAP-B | 패킷 교환망에서 쓰는 데이터링크 프로토콜 |
| LAP-D | ISDN에서 사용 |
| PPP (Point-to-Point Protocol) | 전화회선을 이용한 PC 간 연결 |
| LLC (Logical Link Control) | LAN 프로토콜 / 논리 링크 제어  |

### HDLC 링크 구성

- **HDLC에서 통신하는 장치들이 어떤 관계로 연결되어 있냐**
- HDLC에서는 장치를 그냥 “컴퓨터”라고 안 하고 국(station)이라고 부름.
- **국 종류**
    - **주국** : 통신을 제어하는 쪽
    - **종국** : 주국의 명령에 응답하는 쪽
    - **혼합국** : 명령도 하고 응답도 할 수 있는 쪽
- **링크 구성**
    - **장치들이 어떤 관계인가**
    - **불균형 링크 구성** : 주국이 명령하고 종국이 응답하는 구조
    - **균형 링크 구성** : 양쪽이 서로 통신을 주도할 수 있는 구조 (혼합국 + 혼합국)
- **전송 모드**
    - **정규 응답 모드(NRM, Normal Response Mode)**
        - 주국이 허락해야 종국이 데이터를 보낼 수 있는 방식
        - 불균형
    - **비동기 균형 모드 (ABM, Asynchronous Balanced Mode)**
        - 양쪽이 대등하게 명령도 하고 응답도 할 수 있는 방식
        - 균형
        - 대등한 혼합국 간의 관계
    - **비동기 응답 모드 (ARM, Asynchronous Response Mode)**
        - 불균형
        - 종국이 주국 허락 없이도 데이터를 보낼 수 있는 방식
        - 제어는 주국만 가능

### 응용 계층 프로토콜

1. **HTTP(80) (Hyper Text Transfer Protocol)**
    - **웹 문서**를 주고받는 프로토콜
    - WWW 내 HTML 문서 송수신 위한 프로토콜
    - GET과 POST 메서드를 통해 메시지 주고 받음
2. **FTP(21) (File Transfer Protocol)**
    - **파일**을 전송하는 프로토콜
    - 서버에 파일을 올리거나 내려받을 떄 쓰는 규칙
3. **SMTP(25) (Simple Mail Transfer Protocol)**
    - **이메일**을 보낼 때 프로토콜
4. **DNS (Domain Name System)**
    - **도메인 이름**을 IP 주소로 바꿔주는 프로토콜
5. **DHCP**
    - IP 주소를 **자동으로 할당**해주는 프로토콜
6. **SNMP (Simple Network Management Protocol)**
    - **네트워크 장비**를 관리하는 프로토콜
7. **Telnet(23)**
    - **원격 접속** 프로토콜
    - 멀리 있는 컴퓨터에 접속해서 명령어를 실행하는 방식

### 전송 계층 프로토콜

- **TCP (Transmission Control Protocol)**
    - 데이터를 **정확하게** 보내는 프로토콜
    - **양방향 서비스** 제공
    - 순서 제어, 오류 제어, 흐름 제어 기능 → **높은 신뢰성**
    - 프로토콜 헤더는 기본적으로 **20~60 Byte**
    - 패킷 단위의 스트림 위주 전달
    - UDP보다 **느림**
    - 헤더가 복잡해서 **오버헤드**가 큼
    - 웹 접속, 파일 전송, 이메일처럼 정확해야 하는 통신에 어울림.
- **UDP (User Datagram Protocol)**
    - 데이터를 **빠르게** 보내는 프로토콜
    - 비연결형 서비스 제공
    - TCP 대비 단순한 헤더 구조로 오버헤드 적고, **전송속도 빠름 (제어 X)**
    - 실시간 전송 유리
    - 신뢰성보다는 속도가 중요한 네트워크에 활용
- **RTCP (Real-time Control Protocol)**
    - **실시간 데이터** 전송 상태를 제어하고 품질을 관리하는 프로토콜
    - **RTP** 패킷의 전송 품질 제어

### 네트워크 계층 프로토콜

- **목적지까지 데이터를 보내기 위해 주소를 붙이고, 경로를 찾고, 오류를 알려주는 규칙들**
- **IP (Internet Protocol)**
    - **데이터에 목적지 주소를 붙이고, 목적지까지 갈 수 있게 보내는 프로토콜**
    - 비연결형 데이터그램 방식
    - 신뢰성 보장 X
    - 패킷을 분할 및 병합
    - 헤더 체크섬 제공
    - 데이터 체크섬은 제공 X
- **ICMP (Internet Control Message Protocol)**
    - **IP 통신 중 문제가 생겼을 때 오류나 제어 메시지를 알려주는 프로토콜**
    - 헤더 8Byte
- **ARP (Address Resolution Protocol)**
    - **IP 주소를 MAC 주소로 바꾸는 프로토콜**
    - IP : 네트워크상 목적지 주소
    - MAC : 실제 네트워크 장비의 물리적 주소
- **RARP (Reverse Address Resolution Protocol)**
    - **MAC 주소를 IP 주소로 바꾸는 프로토콜**
- **IGMP (Internet Group Management Protocol)**
    - **멀티캐스트 그룹을 관리하는 프로토콜**
    - 멀티캐스트는 특정 그룹에게만 데이터를 보내는 건데 그때 누가 그 그룹에 속해 있는지, 그룹을 유지하는 역할을 하는 게 IGMP
- **RIP (Routing Information Protocol)**
    - **홉 수가 가장 적은 경로(Hop count 경로, 최단거리 경로)를 선택하는 라우팅 프로토콜**
    - 홉 : 거쳐 가는 라우터 수
- **NAT (Network Address Translation)**
    - **사설 IP를 공인 IP로 바꿔주는 기술**

### IPv4/IPv6 특징

| IPv4 | IPv6 |
| --- | --- |
| 10.97.135.183
(0~255 = 256개 = 8 Bit) | 2544::D6::4B::13F::2C4F
0000~FFFF (각 자리를 콜론으로 구분) |
| 10진수 (0~9) | 16진수 (0~9 + A~F) |
| 8 Bit x 4개 = 32 Bit | 16 Bit x 8개 = 128 Bit |
| 유니, 멀티, 브로드 캐스트 | 유니, 멀티, 애니 캐스트 |
- **IPv6 장점**
    - IPv4보다 주소 확장성, 융통성, 연동성이 높고, 전송 속도가 빠름.
    - 인증성, 기밀성, 데이터 무결성 측면에서 보안 문제 해결 가능
    - 패킷 크기 제한 없으며, 등급/서비스별 패킷 구분 가능 → 품질 보장 용이

### IP 주소 체계

- 32비트의 이진수 → 8비트씩 4개의 필드로 나눈 후 십진수로 표현
- 10101100 00010000 11111110 00000001 → 172.16.254.1

### 캐스팅 전송 방식

- **데이터를 “누구한테 보낼지” 정하는 전송 방식**
- **패킷을 누구에게 보낼지 정하는 방식**
- 패킷 : 보내기 좋게 잘라서 포장한 데이터 조각
- **유니캐스트** : 1 : 1통신
- **멀티캐스트** : 1 : 특정 그룹
- **브로드캐스트** : 네트워크 전체에 전송
- **애니캐스트** : 가장 가까운 한 노드와 통신

### 패킷 교환 방식

- 큰 데이터를 여러 패킷으로 나누고, 네트워크를 통해 목적지까지 보내는 방식
- **가상회선 방식 (VC, Virtual Circuit)**
    - **패킷을 보내기 전에 먼저 길을 정해놓는 방식**
    - 패킷 전송 전 **논리적 통신 경로를 미리 설정**
    - 별도의 가상회선으로 송/수신 간 데이터 전달
    - 전송 완료 후 가상회선도 종료
- **데이터그램 방식 (Datagram)**
    - **각 패킷이 독립적으로 알아서 목적지까지 가는 방식**
    - **연결 경로 사전 설정 없이** 각 패킷을 순서에 무관하게 독립적으로 전송

### 라우팅 (Routing)

- 송신지에서 수신지까지 **패킷이 갈 경로를 선택**하는 것
- **IGP (Interior Gateway Protocol)**
    - **하나의 AS (Autonomous System) 내 라우팅 정보 교환**
    - AS : 하나의 기관이나 조직이 관리하는 네트워크 덩어리
    - **RIP (Routing Information Protocol)**
        - **거쳐 가는 라우터 수가 적은 경로를 선택하는 방식**
        - Hop count
        - 최대 15홉 지원
        - 최단 경로 탐색으로 Bellman-Ford 알고리즘 사용
        - EGP보다는 IGP에 해당
        - 소규모 네트워크 환경 적합
        - **키워드 : Hop count, 거리 벡터, Bellman-Ford, 소규모**
    - **OSPF (Open Shortest Path FIrst Protocol)**
        - 네트워크 상태를 보고 가장 좋은 최단 경로를 찾는 방식
        - RIP 단점 개선 목적
        - 대규모 네트워크에서 널리 사용
        - 실시간 노드 간 거리, 링크 상태 반영
        - 다익스트라 알고리즘 사용
        - **키워드 : 링크 상태, Dijkstra, 대규모, RIP 개선**
- **EGP (Exterior Gateway Protocol)**
    - **서로 다른 AS 간 라우팅 정보를 교환하는 프로토콜**
    - **BGP (Border Gateway Protocol)**
        - 큰 네트워크들끼리 경로를 정할 때 쓰는 라우팅 프로토콜
        - 대규모 네트워크 적합
        - Path Vector 기반 라우팅
        - 다양한 경로 속성 고려한 최적 경로 결정
        - **키워드 : EGP, AS 간, Path Vector, 대규모**

### CIDR 표기법

- IP 주소 뒤에 `/숫자` 붙어 있는 거
- ex)
    
    ```c
    192.168.1.0/24
    192.168.1.0/26
    10.0.0.0/8
    ```
    
    - 여기서 `/24`, `/26`, `/8`
- **IP 주소에서 앞에서 몇 비트까지가 네트워크 부분인지 표시하는 방법**
- IPv4 주소는 총 32비트
근데 이 32비트 안에서 앞부분은 네트워크 주소, 뒷부분은 호스트 주소로 나뉨.
    
    ```c
    [네트워크 부분][호스트 부분]
    ```
    
    CIDR의 `/숫자`는 여기서 네트워크 부분이 몇 비트인지 말함. 
    
- **예시 1: 192.168.1.0/24**
    - 전체 32비트 중 앞 24비트 = 네트워크 부분
    - 나머지 8비트 = 호스트 부분
    - 호스트 부분이 8비트면 만들 수 있는 주소 개수 : 2^8=256개
    - 실제로 쓸 수 있는 호스트 수는 네트워크 주소 1개, 브로드캐스트 주소 1개를 뺌.
    - 사용 가능 호스트 수 = 254개
    - 저렇게 쓰면 이 네트워크 범위는 대충 `192.168.1.0 ~ 192.168.1.255` 라는 걸 알 수 있음.
- **CIDR이랑 서브넷 마스크 관계**
    - **CIDR은 사실 서브넷 마스크를 짧게 쓴 표기법**
    - 예를 들어,
        
        ```c
        /24 = 255.255.255.0
        /25 = 255.255.255.128
        /26 = 255.255.255.192
        /27 = 255.255.255.224
        ```
        
    - 즉, `192.168.1.0/24` 를 풀어서 쓰면
        
        ```c
        IP 주소: 192.168.1.0
        서브넷 마스크: 255.255.255.0
        ```
        

> **CIDR이랑 서브넷 마스크는  “IP들을 몇 개 단위로 묶을 건지 정하는 기준”을 정하는 거임.**

### 정보 전송 방식

- 단방향 : 한쪽 방향으로만 전송 가능 (ex. TV, 라디오)
- 반이중 : 한쪽에서 송신하면 다른 쪽에서는 수신만 가능 (ex. 무전기)
- 전이중 : 동시에 송/수신 가능 (ex. 전화기)

### 네트워크 토폴로지

- 네트워크 장치들이 어떤 형태로 연결되어 있는지 나타낸 구조
- **스타형/성형/중앙 집중형**
    - 중앙노드와 1:1 (P2P) 연결
    - 고장 발견, 유지 보수, 확장이 쉬움
    - 중앙노드의 제어장치가 통신망의 처리능력, 신뢰성 결정
- **버스형**
    - 한 개의 회선에 여러 단말 장치 연결
    - 회선 양 끝 종단장치 필요
    - 단말 장치 하나가 고장나도 전체 영향 없기에 신뢰성 높음
    - 설치/제거 용이
    - **기밀성 낮고 통신 회선의 제한이 있음**
- **링/루프/원/원형**
    - 인접한 단말기를 서로 연결
    - 양향 전송 가능
    - 통신망 하나가 고장 시 전체 통신망 마비
    - **단말 장치 추가/제거, 기밀보호 어려움**
- **트리/계층/분산형**
    - 나뭇가지 모양으로 계층적 연결
    - 분산처리 시스템 구성 방식
    - **확장이 많으면 트래픽 (통신량)이 과중됨**
- **망/매쉬/그물/완전형**
    - 모든 지점의 단말기를 서로 연결
    - 빠른 응답 시간, 높은 연결성
    - 통신량이 많을 경우 유리함 (공중 데이터 통신망)
    - 회선 장애 발생 시 다른 경로로 전송
    - 높은 보안성 및 안정성
    - **단말 장치 추가/제거 어려움 (낮은 확장성)**

### 동기/비동기 전송

|  | 동기식 | 비동기식 |
| --- | --- | --- |
| 전송 단위 | 데이터 블록 단위 | 문자 단위 |
| 동기 제어 방식 | 클럭 동기 | Start bit, Stop bit |
| 통신 속도 | 고속 | 저속 |
| 회로 복잡도 | 복잡 | 단순 |
| 휴지 기간 | 블록 간 휴지 기간 없음
→ 높은 전송 흐름 | 문자 간 휴지 기간 존재 가능
→ 낮은 전송 효율 |
| 예시 | 데이터 통신망, 전화 교환망 | 편지, 이메일 |
| 전송 방식 | 문자 동기 방식 (SYN, STX, ETX)
비트 동기 방식 |  |
- **동기와 비동기 차이**
    - **데이터를 보낼 때 송신자와 수신자가 타이밍을 어떻게 맞추느냐**
    - 핵심 : 언제 데이터가 시작되고 끝나는지 수신자가 어떻게 아냐?
    - **동기** : 송신자와 수신자가 같은 타이밍, 즉 클럭을 맞춰놓고 데이터를 보내는 방식
    - **비동기** : 송신자와 수신자가 계속 박자를 맞추고 있지 않고, 문자마다 시작과 끝을 알려주면서 보내는 방식

### 비동기 전송 모드 (ATM, Asynchronous Transfer Mode)

- 자료를 일정한 크기(Cell)로 정한 후 순서대로 전송
- 멀티미디어 서비스 적합
- Cell : 고정 길이 패킷 크기 **53 Byte** (페이로드 48 Byte + 헤더 5 Byte)
- 단순한 처리 방식 / 고속망 적합 / 실시간 및 비실시간 서비스 제공 가능
- 고정 비트레이트(CBR) 및 가변 비트레이트(VBR) 모두 처리 가능

### 프로토콜 (Protocol)

- 정보 교환을 위해 무엇을, 어떻게, 언제 통신할 것인지(구문, 의미, 순서)를 정한 통신 규약
- **기능**
    - **흐름 제어** : 패킷 흐름(전송량/속도)을 조정하여 송수신 안정성 유지
    - **동기화** : 송/수신이 같은 상태를 유지
    - **오류 제어** : 전송 중 발생한 오류를 검출, 복원/정정

### 흐름제어 방식

- **Stop and Wait**
    - 수신 측 확인 후 다음 패킷 전송
    - 한 번에 하나의 패킷만 전송
- **Sliding Window**
    - 수신 측 확인 없이 사전에 정해진 패킷 수만큼 연속적으로 전송
    - 한 번에 여러 패킷을 동시에 전송 가능
    - 긍정 수신 시 윈도우 크기 증가 / 부정 신호 시 윈도우 크기 감소
- **Slow start**
    - 정상 패킷 전송 시 혼잡 윈도우 크기를 패킷마다 1씩 증가시키나 혼잡 현상 발생 시 혼잡 윈도우 크기를 1로 줄임
- **Congestion Avoidance**
    - 네트워크 내 패킷의 지연이 너무 높아 트래픽이 붕괴되지 않도록 패킷의 흐름을 제어하는 트래픽 제어

### 전송 오류 제어 방식

- **전진 오류 수정 (FEC, Forward Error Correction)**
    - 수신 측에서 **스스로 오류 검출 후 수정까지** 가능한 방식
    - 송신 측에 별도 통보 X
    - ex) 해밍 코드, 상승 코드
- **후진 오류 수정 (BEC, Backward Error Correction)**
    - 오류 발생 시 **송신측에 재전송 요구**
    - ex) 패리티 검사, CRC (Cycle Redundancy Check), 블록 합

### 오류 검출 방식

- **해밍 코드**
    - 수신 측에서 오류 검출 후 자동 수정
    - **1비트 오류 수정** 가능
    - 검출 가능 최대 오류 수 = 해밍 거리 - 1
- **상승 코드**
    - 순차적
    - 한계값 디코딩 사용
    - 여러 비트의 오류 수정 가능
- **패리티 검사**
    - 7~8개 비트로 구성된 **데이터 블록 끝에 특정 패리티 검사 비트(짝수/홀수)**를 추가하여 오류 검출
- **순환 중복 검사 (CRC)**
    - **다항식**을 통해 산출된 값으로 오류 검사
    - 집단 오류 해결
    - 데이터 뒤 오류 검출코드 FCS (Frame Check Sequence) 추가
- **블록 합**
    - 짝수개 비트 오류 검출 불가한 패리티 검사를 개선한 방법
    - 데이터 블록의 수평/수직 패리티 비트 추가

### OAuth

- 다양한 플랫폼의 특정 사용자 데이터에 접근하기 위해 비밀번호를 제공하지 않고 제3자 클라이언트가 사용자의 **접근 권한을 위임** 받을 수 있는 표준 프로토콜
- 사용자 인증에 사용되는 표준 인증 방법
- 공개 API로 구현

### IPC (Inter-Process Communication)

- 실행 프로세스 간 통신을 가능하게 하는 기술
- 종류 : 파이프, 메시지 큐, 공유 메모리, 세마포어, 소켓

### AD-hoc Network

- 기지국/액세스 포인트와 같은 네트워크 장치가 필요하지 않고, 멀티 홉 라우팅 기능에 의해 무선 인터페이스가 가지는 통신 거리상의 제약 극복
- 긴급 구조, 긴급 회의, 전쟁터에서의 군사 네트워크에 활용

### IP 헤더 구조

1. **기본 정보**
    
    ```c
    버전 / 헤더 길이 / 서비스 유형 / 전체 패킷 길이
    ```
    
    - **버전(4)** : IPv4인지 IPv6인지
    - **헤더 길이(4)** : IP 헤더 길이
    - **서비스 유형(8)** : 우선순위, 서비스 품질
    - **전체 길이(16)** : 헤더 + 데이터 전체 길이
2. **단편화 정보**
    
    ```c
    식별자 / 플래그 / 단편 오프셋
    ```
    
    - **식별자(16)** : 이 조각들이 원래 같은 패킷인지 구분하는 번호 / 전송 패킷을 식별하기 위한 부여된 일련번호
    - **플래그(3)** : 분할 가능 여부, 뒤에 조각이 더 있는지 표시 / 단편화(분할) 상태를 표현
    - **단편 오프셋(13)** : 이 조각이 원래 데이터의 몇 번째 위치인지 / 단편화(분할)된 패킷들의 순서 지정
3. **이동/검사 정보**
    
    ```c
    TTL / 프로토콜 / 헤더 체크섬
    ```
    
    - **TTL(8)** : 패킷이 네트워크를 통과할 수 있는 남은 라우터의 최대 개수
    - **프로토콜(8)** : 위에 실린 데이터가 TCP인지 UDP인지 ICMP인지
    - **헤더 체크섬(16)** : IP 헤더에 오류가 있는지 검사
4. **주소 정보**
    - **발신지/목적지 주소** : 패킷을 보낸/받는 IP 주소

### TCP 헤더 구조

1. **송수**
    - 누구 프로그램에서 누구 프로그램으로?
    - **송신지 포트** : 출발지 포트 번호
    - **수신지 포트** : 목적지 포트 번호
2. **일확**
    - 몇 번째 데이터인지, 어디까지 받았는지
    - **일련 번호** : 송신자가 지정한 순서 번호로 몇 번째 데이터인지 전달
    - **확인 응답 번호** : 수신자 → 송신자에게 다음번 데이터가 몇 번째인지 전달
3. **헤예플**
    - **헤더 길이**
    - **예약 필드**
    - **TCP 플래그**
        - TCP 연결 상태를 제어하는 신호
        - URG : 긴급 데이터 존재
        - ACK : 승인 비트
        - PSH : 밀어넣기, 즉각 전송 요구
        - RST : 연결 초기화 비트
        - SYN : 연결 동기화 비트
        - FIN : 연결 종료 비트
4. **윈체긴**
    - 흐름, 오류, 긴급
    - **윈도우 크기** : 한 번에 받을 수 있는 데이터 양
    - **체크섬** : 오류 검사
    - **긴급 포인터** : 긴급 데이터 위치
5. **옵패**
    - **옵션** : 추가 기능
    - **패딩** : 길이 맞추는 채움값

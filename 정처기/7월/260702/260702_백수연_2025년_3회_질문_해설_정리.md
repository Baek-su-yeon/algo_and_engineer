## 5번 — 구조체 포인터와 문자열 포인터 연산

### 문제 핵심

```c
struct Test {
    int i;
    const char *g;
};

struct Test test[] = {{1, "AB"}, {2, "DC"}, {3, "EB"}};
struct Test *p = &test[1];
printf("%s", p->g + (p->i - 1));
```

### 정답

```text
C
```

### 오답 원인

`p->g`를 단순히 문자열 `"DC"` 자체로만 보고, 실제로는 **문자열의 시작 주소**라는 점을 놓쳤다.

### 핵심 풀이

`p = &test[1]`이므로 `p`는 두 번째 구조체를 가리킨다.

```c
test[1] = {2, "DC"}
```

따라서:

```c
p->i == 2
p->g == "DC"의 시작 주소
```

문제의 식은 다음과 같다.

```c
p->g + (p->i - 1)
```

`p->i - 1 = 2 - 1 = 1`이므로:

```c
p->g + 1
```

`p->g`는 문자열 `"DC"`의 첫 글자 `'D'`를 가리키는 주소다.

```text
p->g
 ↓
'D'   'C'   '\0'
[0]   [1]   [2]
```

`p->g + 1`은 한 글자 뒤로 이동하므로 `'C'`를 가리킨다.

```text
      p->g + 1
         ↓
'D'   'C'   '\0'
[0]   [1]   [2]
```

그리고 `%s`는 **해당 주소부터 `\0`을 만날 때까지 문자열로 출력**한다. 따라서 `'C'` 위치부터 출력되어 결과는 `C`가 된다.

### 복습 포인트

```text
p->g          → 문자열 시작 주소
p->g + 1      → 문자열에서 한 글자 뒤 주소
*(p->g + 1)   → 그 주소의 문자
printf("%s", p->g + 1) → 그 주소부터 문자열 출력
```

비슷한 예:

```c
char *s = "KOREA";
printf("%s", s);     // KOREA
printf("%s", s + 1); // OREA
printf("%c", *(s+1)); // O
```

## 7번 — C언어 정수 뒤의 `u` 의미

### 문제 핵심

```c
struct Node {
    struct Node* next;
    unsigned int x;
};

struct Node t1 = { 0, 5u };
struct Node t2 = { 0, 7u };
struct Node t3 = { 0, 11u };

sum = (sum ^ 42u) + 100u;
printf("%u\n", sum);
```

### 정답

```text
187
```

### 질문한 개념

`5u`, `7u`, `11u`, `42u`, `100u`처럼 숫자 뒤에 붙은 `u`가 무엇인지.

### 핵심 개념

`u`는 **unsigned**의 약자다. 정수 뒤에 `u` 또는 `U`를 붙이면 그 정수 상수를 **unsigned int 타입**으로 취급한다.

```c
5    // int 타입 정수 상수
5u   // unsigned int 타입 정수 상수
```

`unsigned int`는 음수를 표현하지 않는 정수형이다.

```text
int          → 음수, 0, 양수 표현 가능
unsigned int → 0, 양수만 표현 가능
```

문제에서는 구조체 멤버가 이미 `unsigned int x`로 선언되어 있다.

```c
unsigned int x;
```

그래서 초기값도 `5u`, `7u`, `11u`처럼 unsigned 상수로 맞춰 쓴 것이다.

### `%u`의 의미

```c
printf("%u", sum);
```

`%u`는 **unsigned int를 출력할 때 쓰는 서식 지정자**다.

```c
printf("%d", x);  // signed int 출력
printf("%u", x);  // unsigned int 출력
```

### 복습 포인트

이번 문제에서는 값들이 모두 양수라서 `u` 때문에 계산 결과가 크게 달라지지는 않는다. 다만 시험에서 다음처럼 보면 된다.

```text
5u, 42u, 100u → unsigned int 상수
%u             → unsigned int 출력
```

`u`는 “부호 없는 정수로 계산하겠다”는 표시다. 음수, 오버플로우, 형 변환이 섞인 문제에서는 결과에 큰 영향을 줄 수 있다.

## 8번 — Java 인터페이스와 `implements`

### 문제 핵심

```java
interface Machine {
    void run();
}

class WashingMachine (____빈칸____) Machine {
    private String name;

    public WashingMachine() {
        this.name = "LG Washer";
    }

    public void run() {
        System.out.println("Washing machine running");
    }
}
```

### 정답

```java
implements
```

완성된 코드:

```java
class WashingMachine implements Machine {
```

### 핵심 개념: interface

`interface`는 클래스가 반드시 지켜야 하는 **약속**이다.

```java
interface Machine {
    void run();
}
```

이 코드는 “Machine이 되려면 반드시 `run()`이라는 메서드를 가져야 한다”는 의미다.

인터페이스 안의 메서드는 보통 실제 동작을 적지 않고 선언만 한다.

```java
void run();
```

즉, 인터페이스는 “이런 기능이 있어야 한다”는 규칙만 정하고, 실제 동작은 구현 클래스가 작성한다.

### 핵심 개념: implements

```java
class WashingMachine implements Machine
```

이 문장은 “WashingMachine 클래스가 Machine 인터페이스의 약속을 지키겠다”는 뜻이다.

따라서 `Machine`이 요구한 `run()`을 반드시 구현해야 한다.

```java
public void run() {
    System.out.println("Washing machine running");
}
```

### `extends`와 `implements` 차이

```text
클래스가 클래스를 상속받을 때      → extends
클래스가 인터페이스를 구현할 때    → implements
```

예시:

```java
class Child extends Parent
```

```java
class WashingMachine implements Machine
```

### 복습 포인트

```text
interface를 클래스에서 사용할 때는 implements
클래스 상속은 extends
인터페이스 구현은 implements
```

## 10번 — SQL `CROSS JOIN`, `LIKE`, `COUNT(*)`

### 문제 핵심

```sql
SELECT COUNT(*) CNT
FROM T1 A
CROSS JOIN T2 B
WHERE A.NAME LIKE B.RULE;
```

T1:

| CODE | NAME |
|---:|---|
| 3258 | smith |
| 4324 | allen |
| 5432 | scott |

T2:

| NO | RULE |
|---:|---|
| 12 | s% |
| 32 | %t% |

### 정답

```text
4
```

### 핵심 개념: CROSS JOIN

`CROSS JOIN`은 두 테이블의 행을 **모든 경우의 수로 조합**한다.

```text
T1의 행 수 × T2의 행 수
```

이 문제에서는:

```text
T1 3행 × T2 2행 = 6행
```

실제 중간 결과는 `NAME`, `RULE`만 생기는 것이 아니라 **T1의 모든 컬럼 + T2의 모든 컬럼**이 모두 포함된다.

| A.CODE | A.NAME | B.NO | B.RULE |
|---:|---|---:|---|
| 3258 | smith | 12 | s% |
| 3258 | smith | 32 | %t% |
| 4324 | allen | 12 | s% |
| 4324 | allen | 32 | %t% |
| 5432 | scott | 12 | s% |
| 5432 | scott | 32 | %t% |

`CODE`와 `NO`는 사라진 것이 아니라, 조건 판단에 필요 없어서 설명에서 생략했을 뿐이다.

### LIKE 조건 적용

```sql
WHERE A.NAME LIKE B.RULE
```

`%`의 의미:

```text
s%   → s로 시작하는 문자열
%t%  → 중간 어디든 t가 들어가는 문자열
```

각 조합을 검사하면:

| A.NAME | B.RULE | 결과 |
|---|---|---|
| smith | s% | O |
| smith | %t% | O |
| allen | s% | X |
| allen | %t% | X |
| scott | s% | O |
| scott | %t% | O |

조건을 만족하는 행은 4개다.

### 최종 결과

```sql
SELECT COUNT(*) CNT
```

`COUNT(*)`는 조건을 만족한 전체 행 수를 세므로 결과는 4다.

### 복습 포인트

```text
JOIN   → 행과 컬럼을 만든다.
WHERE  → 행을 걸러낸다.
SELECT → 최종적으로 보여줄 컬럼을 고른다.
```

만약 다음처럼 조회했다면:

```sql
SELECT A.CODE, A.NAME, B.NO, B.RULE
FROM T1 A
CROSS JOIN T2 B
WHERE A.NAME LIKE B.RULE;
```

결과는 숫자 `4` 하나가 아니라, 조건을 만족한 4개의 행이 출력된다.

## 14번 — 관계대수 Division 연산과 `%` 표기

### 문제 핵심

문제에서는 `R % S` 형태로 표기되어 있지만, 설명에 “나누기, division”이라고 되어 있으므로 관계대수의 **Division 연산**을 묻는 문제다.

### 정답

| A |
|---|
| a1 |

### 핵심 개념: Division

관계대수의 Division은 보통 다음 기호로 쓴다.

```text
R ÷ S
```

의미는 다음과 같다.

```text
S에 있는 모든 값과 짝을 이루는 R의 값을 찾아라.
```

예를 들어:

| 학생 | 과목 |
|---|---|
| 철수 | DB |
| 철수 | OS |
| 영희 | DB |

S 테이블:

| 과목 |
|---|
| DB |
| OS |

`R ÷ S`는 “S에 있는 모든 과목을 수강한 학생”을 찾는다.

철수는 DB와 OS를 모두 수강했으므로 통과한다. 영희는 DB만 있고 OS가 없으므로 탈락한다.

결과:

| 학생 |
|---|
| 철수 |

### `%`는 나머지 연산이 아님

이 문제에서 `%`는 C언어, Java, Python의 나머지 연산자가 아니다.

```text
C/Java/Python의 %       → 나머지 연산
SQL LIKE의 %           → 문자열 와일드카드
이 문제의 R % S 표기    → 관계대수 Division 의미
```

즉, `R % S`를 보고 산술적으로 나머지를 구하는 것이 아니라:

```text
R ÷ S
```

처럼 해석해서 Division 결과를 구해야 한다.

### 복습 포인트

Division은 “하나라도 만족”이 아니라 **모든 값을 만족**해야 한다.

```text
S의 값이 b1, b2라면
A가 결과에 남으려면 R 안에 (A, b1), (A, b2)가 모두 있어야 한다.
```

이번 문제에서는 `S`의 모든 값을 만족하는 `A`가 `a1`뿐이므로 결과는 `a1`이다.

## 15번 — C 연산자 우선순위 정리

### 문제 핵심

```c
int x=7, y=4, z;
z = y%3<3 ? 2 : 1;
z = z & z >> 1;
z = x>5 && z<=3 ? z*x : z/x;
printf("%d", z);
```

### 정답

```text
0
```

### C 연산자 우선순위 표

위에 있을수록 먼저 계산된다.

| 우선순위 | 연산자 | 의미 | 결합 방향 |
|---:|---|---|---|
| 1 | `()`, `[]`, `->`, `.`, `expr++`, `expr--` | 괄호, 배열, 구조체 접근, 후위 증감 | 왼쪽 → 오른쪽 |
| 2 | `++expr`, `--expr`, 단항 `+`, 단항 `-`, `!`, `~`, `*`, `&`, `sizeof` | 전위 증감, 단항 연산, 역참조, 주소 | 오른쪽 → 왼쪽 |
| 3 | `*`, `/`, `%` | 곱셈, 나눗셈, 나머지 | 왼쪽 → 오른쪽 |
| 4 | `+`, `-` | 덧셈, 뺄셈 | 왼쪽 → 오른쪽 |
| 5 | `<<`, `>>` | 비트 시프트 | 왼쪽 → 오른쪽 |
| 6 | `<`, `<=`, `>`, `>=` | 대소 비교 | 왼쪽 → 오른쪽 |
| 7 | `==`, `!=` | 같음, 다름 비교 | 왼쪽 → 오른쪽 |
| 8 | `&` | 비트 AND | 왼쪽 → 오른쪽 |
| 9 | `^` | 비트 XOR | 왼쪽 → 오른쪽 |
| 10 | `|` | 비트 OR | 왼쪽 → 오른쪽 |
| 11 | `&&` | 논리 AND | 왼쪽 → 오른쪽 |
| 12 | `||` | 논리 OR | 왼쪽 → 오른쪽 |
| 13 | `?:` | 삼항 연산자 | 오른쪽 → 왼쪽 |
| 14 | `=`, `+=`, `-=`, `*=`, `/=`, `%=` 등 | 대입 연산자 | 오른쪽 → 왼쪽 |
| 15 | `,` | 콤마 연산자 | 왼쪽 → 오른쪽 |

### 문제 풀이

#### 1줄

```c
z = y%3<3 ? 2 : 1;
```

`%`가 `<`보다 먼저다.

```text
y % 3 = 4 % 3 = 1
1 < 3 → 참
z = 2
```

#### 2줄

```c
z = z & z >> 1;
```

시프트 `>>`가 비트 AND `&`보다 먼저다.

```text
z = 2
z >> 1 = 1
2 & 1 = 10₂ & 01₂ = 00₂ = 0
z = 0
```

#### 3줄

```c
z = x>5 && z<=3 ? z*x : z/x;
```

비교 연산이 먼저, 그다음 `&&`, 그다음 삼항 연산자다.

```text
x > 5  → 7 > 5 → 참
z <= 3 → 0 <= 3 → 참
참 && 참 → 참
```

삼항 연산자의 조건이 참이므로:

```text
z * x = 0 * 7 = 0
```

출력은 `0`이다.

### 복습 포인트

시험장에서 전체 표가 기억나지 않으면 최소한 다음 순서는 기억한다.

```text
괄호/배열/구조체
→ 단항
→ 산술(* / %)
→ 산술(+ -)
→ 시프트
→ 비교
→ 동등
→ 비트 AND
→ 비트 XOR
→ 비트 OR
→ 논리 AND
→ 논리 OR
→ 삼항
→ 대입
```

짧게 압축하면:

```text
산술 > 비교 > 동등 > 비트 > 논리 > 삼항 > 대입
```

## 16번 — 스키마의 의미

### 질문한 개념

16번에서 정답 자체보다, 보기로 나온 **스키마**가 무슨 뜻인지 질문했다.

### 핵심 개념

스키마는 쉽게 말하면 **데이터베이스의 설계도**다.

데이터 자체가 아니라, 데이터를 어떤 구조로 저장할지 정해 놓은 틀이다.

예를 들어 학생 테이블이 있다고 하자.

| 학번 | 이름 | 학과 | 학년 |
|---|---|---|---|
| 2025001 | 김철수 | 컴퓨터공학과 | 3 |
| 2025002 | 이영희 | 정보보호학과 | 2 |

실제 데이터는 다음이다.

```text
2025001, 김철수, 컴퓨터공학과, 3
2025002, 이영희, 정보보호학과, 2
```

반면 스키마는 다음과 같은 구조를 말한다.

```text
학생(학번, 이름, 학과, 학년)
```

조금 더 자세히 말하면:

```text
학생 테이블은 학번, 이름, 학과, 학년이라는 속성을 가진다.
학번은 기본키다.
이름은 문자열이다.
학년은 숫자다.
```

이런 식으로 테이블의 구조, 속성, 제약조건 등을 정의한 것이 스키마다.

### 인스턴스와 비교

```text
스키마   → DB의 구조/설계도
인스턴스 → 어느 순간 실제로 저장된 데이터 전체
```

예시:

```text
스키마: 학생(학번, 이름, 학과, 학년)
인스턴스: 실제 학생 데이터들이 들어 있는 현재 상태
```

### 복습 포인트

```text
스키마 = 틀
인스턴스 = 그 틀 안에 들어간 실제 데이터
```

## 17번 — Java `enum`, `values()`, `name()`

### 문제 핵심

```java
enum Tri {
    A("A"), B("AB"), C("ABC");

    private String code;

    Tri(String code) {
        this.code = code;
    }

    public String code() {
        return code;
    }
}

public class Main {
    public static void main(String[] args) {
        Tri t = Tri.values()[Tri.A.name().length()];
        System.out.print(t.code());
    }
}
```

### 정답

```text
AB
```

### 핵심 개념: enum

`enum`은 **정해진 값들만 가질 수 있는 자료형**이다.

```java
enum Tri {
    A("A"), B("AB"), C("ABC");
}
```

`Tri`가 가질 수 있는 값은 딱 세 개다.

```text
Tri.A
Tri.B
Tri.C
```

### enum 상수에 추가 값 붙이기

```java
A("A"), B("AB"), C("ABC");
```

각 enum 상수에 `code` 값을 붙인 것이다.

| enum 값 | code 값 |
|---|---|
| `A` | `"A"` |
| `B` | `"AB"` |
| `C` | `"ABC"` |

이 값을 저장하는 필드가 다음이다.

```java
private String code;
```

생성자에서 값을 저장한다.

```java
Tri(String code) {
    this.code = code;
}
```

`code()` 메서드는 저장된 값을 반환한다.

```java
public String code() {
    return code;
}
```

따라서:

```text
Tri.A.code() → "A"
Tri.B.code() → "AB"
Tri.C.code() → "ABC"
```

### `values()`

`values()`는 enum이 자동으로 제공하는 메서드다. enum 상수들을 배열로 반환한다.

```java
Tri.values()
```

결과는 다음 배열처럼 볼 수 있다.

```text
[Tri.A, Tri.B, Tri.C]
```

인덱스로 보면:

| 인덱스 | 값 |
|---:|---|
| 0 | `Tri.A` |
| 1 | `Tri.B` |
| 2 | `Tri.C` |

### `name()`

`name()`은 enum 상수의 이름 자체를 문자열로 반환한다.

```java
Tri.A.name() → "A"
Tri.B.name() → "B"
Tri.C.name() → "C"
```

주의할 점은 `name()`과 `code()`가 다르다는 것이다.

```text
Tri.B.name() → "B"
Tri.B.code() → "AB"
```

### 문제 풀이

핵심 줄:

```java
Tri t = Tri.values()[Tri.A.name().length()];
```

오른쪽부터 계산한다.

```text
Tri.A.name() → "A"
"A".length() → 1
Tri.values()[1] → Tri.B
t = Tri.B
```

마지막 출력:

```java
System.out.print(t.code());
```

`t`는 `Tri.B`이고, `Tri.B.code()`는 `"AB"`다.

따라서 출력은:

```text
AB
```

### 복습 포인트

```text
enum 값들은 values()로 배열처럼 꺼낼 수 있다.
name()은 enum 상수 이름을 문자열로 반환한다.
생성자를 두면 enum 상수마다 추가 값을 저장할 수 있다.
```

## 20번 — SQL `OR` 조건과 `COUNT(col2)`의 NULL 제외

### 문제 핵심

```sql
SELECT count(col2)
FROM A
WHERE col1 IN (2, 3) OR col2 IN (3, 5);
```

A 테이블:

| 행 | col1 | col2 |
|---:|---:|---:|
| 1 | 2 | NULL |
| 2 | 3 | 6 |
| 3 | 2 | 3 |
| 4 | NULL | 3 |
| 5 | 4 | 5 |

### 정답

```text
4
```

### 헷갈린 지점

조건이 `OR`로 연결되어 있으므로 둘 중 하나라도 만족하면 선택된다. 그래서 WHERE 조건만 보면 5개 행이 모두 선택되는 것처럼 보인다.

이 판단은 맞다.

### WHERE 조건 확인

```sql
WHERE col1 IN (2, 3) OR col2 IN (3, 5)
```

| 행 | col1 IN (2,3) | col2 IN (3,5) | OR 결과 | 선택 여부 |
|---:|---|---|---|---|
| 1 | 참 | NULL은 3/5 아님 | 참 | 선택 |
| 2 | 참 | 6은 3/5 아님 | 참 | 선택 |
| 3 | 참 | 참 | 참 | 선택 |
| 4 | NULL은 2/3 아님 | 참 | 참 | 선택 |
| 5 | 4는 2/3 아님 | 참 | 참 | 선택 |

따라서 WHERE 조건을 만족하는 행은 5개다.

### 그런데 왜 결과가 4인가?

최종 집계가 `COUNT(*)`가 아니라 `COUNT(col2)`이기 때문이다.

```sql
SELECT count(col2)
```

`COUNT(col2)`는 선택된 행 중에서 **col2가 NULL인 행은 세지 않는다**.

선택된 5개 행의 `col2`만 보면:

| 행 | col2 | COUNT(col2)에 포함? |
|---:|---:|---|
| 1 | NULL | 제외 |
| 2 | 6 | 포함 |
| 3 | 3 | 포함 |
| 4 | 3 | 포함 |
| 5 | 5 | 포함 |

따라서 실제로 세는 값은 다음 4개다.

```text
6, 3, 3, 5
```

결과는 4다.

### 복습 포인트

```text
WHERE 조건을 만족하는 행 수 = 5개
COUNT(*)였다면 결과 = 5
COUNT(col2)이므로 col2가 NULL인 행 제외 = 4
```

SQL 집계 함수에서 자주 나오는 차이:

```text
COUNT(*)      → NULL 포함, 선택된 전체 행 수
COUNT(컬럼명) → 해당 컬럼이 NULL이 아닌 행 수
```

이번 문제의 함정은 `OR`가 아니라 **COUNT(컬럼명)은 NULL을 세지 않는다**는 점이다.
